# critique-guide.md —— 5 維度設計評審

**何時載入**：做完 hi-fi 後自我評審、Mode 3（優化既有設計）時、使用者說「看看有沒有哪裡可以改」、或送交前的品質守門。這份文件提供**結構化的評審框架**，避免憑感覺說「好像還不錯」就交付。

---

## 為什麼要做評審

AI 最常犯的錯：做完一個「看起來 OK 的東西」就交付，然後使用者回饋「哪裡怪怪的但說不上來」。這種模糊反饋通常來自**沒被顯化的品質問題**——版面層級不清、間距不一致、色彩沒系統、裝飾蓋過內容。

5 維度評審的作用：**把「感覺怪」變成「具體哪裡怪」**，讓您能精準修、讓使用者能精準說。

**操作時機**：
- Step 5 variations 做完後自評，挑出最好的一版
- Step 6 交付前的最後一關
- 使用者給回饋「再改改看」時，您主動做一次評審提議具體方向
- 收到別人做的設計請您「看看怎麼樣」時

---

## 5 個評審維度

### 1. 哲學一致性（Philosophy Alignment）

**問題**：這份設計有沒有落在選定的 design-philosophy 風格裡？還是中途跑掉了？

| 分數 | 標準 |
|------|------|
| 9-10 | 完美體現選定風格，每個細節都有哲學依據 |
| 7-8 | 方向正確、核心特徵到位、個別細節偏離 |
| 5-6 | 能看出意圖，但混入了其他風格元素 |
| 3-4 | 表面模仿、未理解哲學內核 |
| 1-2 | 與選定風格基本無關 |

**評審要點**：
- 選了「經濟日報社論版」卻出現紫漸變和 emoji —— 自相矛盾
- 選了「顧問諮詢所」卻用圓角 + 左色條 —— 混入 SaaS 味
- 選了「日式極簡」卻把畫面填滿 —— 違反風格核心
- 色彩、字型、layout 是否符合該風格的視覺語言？

---

### 2. 視覺層級（Visual Hierarchy）

**問題**：使用者的視線能自然沿您意圖流動嗎？還是平鋪一片不知道先看哪裡？

| 分數 | 標準 |
|------|------|
| 9-10 | 視線自然流動、零摩擦獲取資訊 |
| 7-8 | 主次清晰、偶有 1-2 處層級模糊 |
| 5-6 | 能分標題內文、中間層級混亂 |
| 3-4 | 信息平鋪、沒有視覺入口 |
| 1-2 | 混亂、不知道先看哪 |

**評審要點**：
- 標題與正文字級對比是否足夠（**至少 2.5 倍**，通常 3 倍以上）？
- 顏色 / 粗細 / 大小是否建立 3-4 個清晰層級？
- 留白是否引導視線？
- **瞇眼測試**：瞇起眼看，層級是否仍然清晰？看得到的是標題、主數據、還是背景？

---

### 3. 細節執行（Craft Quality）

**問題**：像素級精確嗎？對齊、間距、顏色有沒有系統化？

| 分數 | 標準 |
|------|------|
| 9-10 | 像素級精確、無瑕疵 |
| 7-8 | 整體精緻、1-2 處微小對齊或間距問題 |
| 5-6 | 基本對齊、間距不統一、顏色不夠系統 |
| 3-4 | 明顯對齊錯誤、間距混亂、顏色過多 |
| 1-2 | 粗糙、像草稿 |

**評審要點**：
- 使用**統一的間距系統**（8pt 網格：8/16/24/32/48/64）？
- 同類元素的間距**一致**？
- 顏色數量**受控**（通常不超過 3-4 種）？
- 字體家族**統一**（通常不超過 2 種）？
- 邊緣對齊**精確**（不是「看起來差不多」）？

---

### 4. 功能性（Functionality）

**問題**：每個元素都服務於目標嗎？還是為了好看硬加的？

| 分數 | 標準 |
|------|------|
| 9-10 | 每個元素都服務目標、零冗餘 |
| 7-8 | 功能明確、有少量可刪減的裝飾 |
| 5-6 | 基本可用、有裝飾性元素分散注意力 |
| 3-4 | 形式大於功能、使用者要努力找資訊 |
| 1-2 | 完全被裝飾淹沒、失去傳達能力 |

**評審要點**：
- **刪掉任一元素**，設計會變差嗎？如果不會，就應該刪
- CTA / 關鍵資訊是否在最顯眼的位置？
- 有沒有「因為好看所以加上去」的元素？
- 信息密度與載體是否匹配（PPT 不宜太密、PDF 可以更密）？

---

### 5. 創新性（Originality）

**問題**：在該哲學框架內找到了獨特表達嗎？還是模板套用？

| 分數 | 標準 |
|------|------|
| 9-10 | 令人耳目一新、在框架內獨特表達 |
| 7-8 | 有自己的想法、非簡單模板套用 |
| 5-6 | 中規中矩、像模板 |
| 3-4 | 大量使用 cliché |
| 1-2 | 完全是模板或素材拼湊 |

**評審要點**：
- 避免了常見 cliché？（紫漸變、圓角 + 左色條、emoji、bento 泛濫、大 Hero + 3 欄 features）
- 在遵循設計哲學的同時有個人表達？
- 有沒有「意想不到但很合理」的設計決策？

**注意**：對**政府標案、月報、法規文件**這類場景，**創新性可以放寬**——清晰 / 合規 / 完整比創新重要。

## 場景評審的側重點（台灣中小企業版）

不同輸出類型的評審重點不同。用這張表挑哪些維度最重要：

| 場景 | 最重要 | 次重要 | 可放寬 |
|------|--------|--------|--------|
| 競品 battlecard（內部） | 視覺層級、功能性 | 細節執行 | 創新性 |
| 客戶研究 brief（業務用） | 功能性、細節執行 | 視覺層級 | 創新性 |
| 月報（給老闆看） | 功能性、視覺層級 | 細節執行 | 創新性 |
| 品牌故事一頁式 | 哲學一致性、創新性 | 視覺層級 | 功能性 |
| 對外提案簡報 | 視覺層級、功能性 | 哲學一致性 | 創新性 |
| 政府補助申請書 | 功能性、細節執行 | 哲學一致性（合規 = 哲學） | 創新性（禁止） |
| PMM 深度分析 | 功能性、細節執行 | 視覺層級 | 創新性 |
| 一對一客戶 deck | 哲學一致性、視覺層級 | 功能性 | 創新性 |

## 台灣場景的常見設計問題 Top 10

### 1. 字級層級不足（最常見）

**問題**：標題和正文差距太小（< 2.5 倍）
**修復**：標題至少 3 倍正文（例：正文 16px → 標題 48-64px）

### 2. 顏色過多

**問題**：5 種以上顏色，沒有主次
**修復**：限制為 1 主色 + 1 輔色 + 1 強調色 + 灰階

### 3. 間距不統一

**問題**：元素間距隨意
**修復**：用 8pt 網格（只用 8/16/24/32/48/64px）

### 4. 留白不足

**問題**：所有空間都被內容填滿，看起來擠
**修復**：留白至少佔 40%，極簡風格 60%+

### 5. 字型過多

**問題**：3 種以上字型
**修復**：最多 2 種（1 標題 + 1 正文），用字重和大小創造變化

### 6. 對齊不一致

**問題**：有的左對齊、有的置中、有的右對齊
**修復**：選一種（推薦左對齊），全局統一

### 7. 裝飾大於內容

**問題**：背景圖案 / 漸變 / 陰影搶了內容的風頭
**修復**：「刪掉這個裝飾，設計會變差嗎？」如果不會就刪

### 8. 信息密度與載體不匹配

- PPT 每頁 1 個核心觀點（塞太多是地雷）
- 封面圖 1 個視覺焦點
- PDF 可以更密，但需要清晰導航
- LINE 轉傳用縮圖也要看得清

### 9. 紫 / 漸變 cliché

**問題**：AI 默認配色一眼就是 AI 味
**修復**：參考 design-philosophy 選實戰驗證的配色

### 10. 沒對齊品牌色 / 字型

**問題**：憑記憶或猜一個「大概的」
**修復**：按 asset-protocol.md 的 5 步流程抓真實品牌色

## 評審輸出模板

交付評審時用這個結構（可以自評、也可以回饋別人的設計）：

```markdown
## 設計評審 —— {檔案名}

**總體**：X.X / 10 [優秀 8+ / 良好 6-7.9 / 需改進 4-5.9 / 不合格 <4]

**分項**：
- 哲學一致性：X/10 —— [一句話]
- 視覺層級：X/10 —— [一句話]
- 細節執行：X/10 —— [一句話]
- 功能性：X/10 —— [一句話]
- 創新性：X/10 —— [一句話]（此維度在本場景可放寬 / 嚴格）

### 優點（Keep）

- [具體指出做得好的地方，用設計語言描述]

### 問題（Fix，按嚴重度排序）

**1. [問題名稱]** —— ⚠️ 致命 / ⚡ 重要 / 💡 優化
- 當前：[描述現狀]
- 問題：[為什麼是問題]
- 修復：[具體操作、含數值]

### 快速修復清單（5 分鐘可以做完的）

- [ ] [最有影響力的修復]
- [ ] [第二重要的]
- [ ] [第三重要的]
```

## 跟使用者交付評審結果的語氣

- **對主管**：「我自評了一下，第 X 項可能要再調，給您看看」
- **給年輕員工回饋**：可以直接說「這裡層級不夠清楚，標題要放大到 3 倍內文」
- **送交使用者審閱前的自我檢查**：不用跟使用者說「我做了評審」，直接把修正好的版本給他

**不要做的**：交付時寫一份 200 字的評分報告。評審是**給您自己用的工具**，不是給使用者看的儀式。使用者只要看到結果品質提升即可。

## 快速參考

1. **5 維度**：哲學一致性 / 視覺層級 / 細節執行 / 功能性 / 創新性
2. **不同場景、不同側重**：月報重功能、品牌故事重創新
3. **台灣常見 10 地雷**：字級層級、顏色過多、間距不統一、留白不足、字型過多 ...
4. **瞇眼測試**：瞇起眼看層級還在不在
5. **交付評審不要做成儀式**：自評是工具，不是報告
