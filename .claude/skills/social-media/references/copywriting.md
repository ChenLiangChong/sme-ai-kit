## copywriting


# 文案撰寫

本模組負責轉換型文案撰寫，目標是產出清楚、有說服力、能驅動行動的行銷文案。

## 撰寫前準備

**先檢查既有的品牌／行銷 context：**
在提問之前，先用 `query_knowledge(category='brand')` 和 `query_knowledge(category='marketing')` 檢查既有 context。

蒐集以下 context（若未提供請詢問）：

### 1. 頁面目的
- 這是哪一種頁面？（首頁、Landing Page（到達頁／著陸頁）、定價頁、功能頁、關於頁）
- 您希望訪客採取的**那一個**主要行動是什麼？

### 2. 受眾
- 理想客戶是誰？
- 他們想解決什麼問題？
- 他們有哪些異議或猶豫？
- 他們會用什麼樣的語言描述自己的問題？

### 3. 產品／提案
- 您在賣什麼或提供什麼？
- 與替代方案相比，什麼讓它不一樣？
- 關鍵的轉變或成果是什麼？
- 有沒有證據點（數字、客戶見證、案例研究）？

### 4. Context
- 流量從哪來？（廣告、自然流量、Email）
- 訪客抵達前已經知道什麼？


## 文案撰寫原則

### 清楚勝過聰明
如果必須在「清楚」與「有創意」之間選一個，選清楚。

### 好處勝過功能
功能：它在做什麼。好處：那對客戶代表什麼意義。

### 具體勝過含糊
- 含糊：「為您的工作流程節省時間」
- 具體：「把每週報表時間從 4 小時縮短為 15 分鐘」

### 客戶的語言勝過公司的語言
用客戶在用的字詞。從評論、訪談、客服單中鏡射客戶之聲。

### 一段一個主張
每一段只推進一個論點。在頁面上建立邏輯流。


## 寫作風格規則

### 核心原則

1. **簡單勝過複雜** —— 用「use」不用「utilize」，用「help」不用「facilitate」
2. **具體勝過含糊** —— 避免「streamline」「optimize」「innovative」
3. **主動勝過被動** —— 「我們產出報表」而不是「報表被產出」
4. **肯定勝過修飾** —— 刪掉「almost」「very」「really」
5. **呈現勝過告知** —— 描述結果，而不是用副詞
6. **誠實勝過聳動** —— 絕對不編造統計或客戶見證

### 快速品質檢查

- 有沒有會讓外行人困惑的行話？
- 句子是不是想做太多事？
- 有沒有被動語態的結構？
- 有沒有驚嘆號？（刪掉）
- 有沒有空洞的行銷流行語？

若要逐行深入檢視，請在擬稿後使用 **copy-editing** 技能。


## 最佳實踐

### 直接
直接講重點，不要把價值埋在修飾語裡。

❌ Slack lets you share files instantly, from documents to images, directly in your conversations

✅ Need to share a screenshot? Send as many documents, images, and audio files as your heart desires.

### 使用反問句
問句能吸引讀者並讓他們思考自己的狀況。
- 「討厭退貨給 Amazon 嗎？」
- 「老是在追著跑簽核嗎？」

### 必要時善用類比
類比能讓抽象概念變具體、好記。

### 適時加入幽默
雙關與機智讓文案令人難忘—— 但只限於符合品牌、不會傷害清楚度的情況。


## 頁面結構框架

### 首屏 (Above the Fold — 用戶無需滾動即可見的範圍)

**主標（Headline）**
- 您最重要的那一句話
- 傳達核心價值主張
- 具體 > 通用

**範例公式：**
- 「{達成結果}，不必 {痛點}」
- 「給 {受眾} 的 {品類}」
- 「再也不用 {不愉快的事}」
- 「{點出核心痛點的問句}」


**副標（Subheadline）**
- 展開主標
- 增加具體度
- 最多 1-2 句

**主要 CTA**
- 動作導向的按鈕文字
- 傳達對方會得到什麼：「開始免費試用」> 「註冊」

### 核心段落

| 段落 | 目的 |
|---------|---------|
| 社群背書 (Social Proof) | 建立可信度（客戶 logo、數據、客戶見證） |
| 問題／痛點 | 讓對方知道您理解他們的處境 |
| 解法／好處 | 連結到結果（3-5 個關鍵好處） |
| 運作方式 | 降低感受到的複雜度（3-4 個步驟） |
| 異議處理 | FAQ、比較、保證 |
| 最終 CTA | 重述價值、再喊一次 CTA、反向風險保證 |


## CTA 文案指引

**弱 CTA（避免使用）：**
- Submit、Sign Up、Learn More、Click Here、Get Started

**強 CTA（建議使用）：**
- Start Free Trial
- Get [Specific Thing]
- See [Product] in Action
- Create Your First [Thing]
- Download the Guide

**公式：** [動詞] + [對方會得到什麼] + [必要修飾語]

範例：
- 「Start My Free Trial」
- 「Get the Complete Checklist」
- 「See Pricing for My Team」


## 各類頁面的指引

### 首頁
- 服務多種受眾，但不要淪於通用
- 以最廣義的價值主張領頭
- 為不同訪客意圖提供明確路徑

### Landing Page
- 單一訊息，單一 CTA
- 主標要呼應廣告／流量來源
- 在單一頁面內完成完整論證

### 定價頁
- 協助訪客選到適合的方案
- 化解「哪一個適合我？」的焦慮
- 讓推薦方案一眼可見

### 功能頁
- 連結功能 → 好處 → 成果
- 展示使用情境與範例
- 清楚的試用或購買路徑

### 關於頁
- 講述您為什麼存在的故事
- 把使命連結到客戶受益
- 仍要放上 CTA


## 語氣（Voice and Tone）

撰寫前先確立：

**正式度：**
- 輕鬆／口語
- 專業但親切
- 正式／企業級

**品牌人格：**
- 俏皮還是嚴肅？
- 大膽還是低調？
- 技術取向還是平易近人？

維持一致，但可調整強度：
- 主標可以更大膽
- 內文要更清楚
- CTA 要動作導向


## 產出格式

撰寫文案時請提供：

### 頁面文案
依段落組織：
- 主標 (Headline)、副標 (Subheadline)、行動呼籲 (CTA)
- 段落標題與內文
- 次要 CTA

### 註記
針對關鍵元素，說明：
- 為什麼這樣選
- 套用了哪一條原則

### 替代方案
主標與 CTA 請提供 2-3 個選項：
- 方案 A：[文案] —— [理由]
- 方案 B：[文案] —— [理由]

### Meta 內容（若相關）
- 頁面標題（for SEO）
- Meta description


## Proactive Triggers（主動觸發）

在 context 中察覺以下問題時，**不等使用者問**就提出：

- **文案以「我們」或公司名開頭** → 立刻提醒；改寫為先帶客戶的結果或痛點。
- **價值主張含糊**（例如「給團隊用的最佳平台」）→ 要求具體化：對象是誰、什麼成果、花多久。
- **只列功能沒寫好處** → 交稿前在功能後加上「which means...」橋接到好處。
- **沒有任何社群背書** → 標示為轉換風險，並向使用者索取客戶見證、數據或案例。
- **CTA 使用弱動詞**（Submit、Learn More、Sign Up）→ 在定稿前提出動作＋成果的替代方案。


## Output Artifacts（產出物）

| 使用者要什麼 | Claude 給什麼 |
|---------------------|------------|
| 首頁文案 | 按段落組織的完整頁面文案：主標、副標、CTA、社群背書、好處、運作方式、異議處理、最終 CTA |
| Landing Page | 單一焦點的文案，含主標、內文、一個 CTA —— 附轉換理由註記 |
| 主標選項 | 5 種主標變體，使用不同公式（成果型、痛點型、問句型、大膽宣言、分類型） |
| CTA 文案 | 3-5 個 CTA 選項，每個附公式與理由 |
| 頁面文案審閱 | 逐段針對清楚度、好處框架、CTA 強度的回饋 |


## Communication（溝通原則）

所有產出遵循結構化溝通標準：

- **先結論** —— 先交出文案，再說明選擇背後的理由
- **What + Why + How** —— 每個文案決定都有背後原則
- **註記必要** —— 交稿前不可沒有附上關鍵選擇的說明
- **信心標註** —— 🟢 強烈建議 / 🟡 需要測試 / 🔴 需要證據才站得住腳

高風險元素（主標、CTA）一律提供替代方案。絕不給單一選項就交差。

## 台灣中文文案注意事項

### 繁簡差異

台灣使用繁體中文，用語與簡體中文（中國大陸）有明顯差異。錯用簡體用語會降低品牌信任感。

| 台灣繁體（正確） | 簡體/大陸用語（避免） | 說明 |
|----------------|-------------------|------|
| 按讚 | 點贊 | 社群互動 |
| 貼文 | 帖子 | 社群內容 |
| 分享 | 轉發 | 內容擴散 |
| 追蹤 | 關注 | 社群追蹤 |
| 網紅 | 博主、大V | KOL 稱呼 |
| 影片 | 視頻 | 影音內容 |
| 軟體 | 軟件 | 技術用語 |
| 簡訊 | 短信 | 通訊用語 |
| 部落格 | 博客 | Blog |
| 優惠券 | 優惠券/卡券 | 促銷工具 |
| 限時動態 | 快拍 | Stories |

### 語氣與口吻

- **親切但不隨便** —— 台灣消費者偏好友善、有人情味的溝通，但過度輕浮會失去信任
- **「您」與「你」** —— 對客戶用「您」表尊重，品牌人格化內容可用「你」拉近距離
- **避免過度浮誇** —— 「全台最強」「史上最低」等誇大用語易被消費者質疑，且可能觸法
- **在地感** —— 適當使用台灣口語（「超推」「必收」「懶人包」）能提升親和力
- **Emoji 適量** —— 台灣社群文案常用 Emoji 增加可讀性，但不宜過多影響專業感

### CTA 用語

| 英文 CTA | 台灣常用中文 CTA | 說明 |
|----------|----------------|------|
| Buy Now | 立即購買、馬上下單 | 購物轉換 |
| Learn More | 了解更多、看更多 | 資訊導向 |
| Sign Up | 立即加入、免費註冊 | 註冊導向 |
| Book Now | 立即預約、馬上預約 | 服務業 |
| Get Started | 開始使用、免費試用 | 軟體/工具 |
| Shop Now | 逛逛看、去逛逛 | 瀏覽導向（較軟性） |
| Add to Cart | 加入購物車 | 電商 |
| Claim Offer | 領取優惠、立即領取 | 促銷 |
| Contact Us | 聯絡我們、線上諮詢 | 服務/B2B |
| Add Friend | 加入好友 | LINE 導流 |

### 節慶文案注意事項

- **農曆新年** —— 避免不吉利用語，善用「福」「旺」「好運」等吉祥字
- **母親節/父親節** —— 感恩主題，避免過度商業化
- **雙 11/雙 12** —— 強調折扣力度、限時搶購、數字呈現
- **中秋節** —— 團圓、送禮主題，企業送禮需求大

### 法規紅線

- **不得虛偽不實** —— 「全台第一」「最低價」等須有依據
- **促銷條件須明示** —— 限時限量需標明具體時間和數量
- **KOL 合作須揭示** —— 合作文須標示 #合作 #廣告
- **個資蒐集須告知** —— 收集 Email、LINE ID 等須說明用途

## content-strategy


# 內容策略

本模組負責內容策略規劃，目標是協助規劃能帶來流量、建立權威、產出線索的內容，做到「可搜尋」、「可分享」，或兩者兼具。

## 規劃前準備

**先檢查既有的品牌／行銷 context：**
在提問之前，先用 `query_knowledge(category='brand')` 和 `query_knowledge(category='marketing')` 檢查既有 context。

蒐集以下 context（若未提供請詢問）：

### 1. 商業 Context
- 公司在做什麼？
- 理想客戶是誰？
- 內容的首要目標是什麼？（流量、線索、品牌知名度、思想領導）
- 產品解決什麼問題？

### 2. 客戶研究
- 客戶購買前會問什麼問題？
- 業務電話中常出現哪些異議？
- 客服單中反覆出現什麼主題？
- 客戶怎麼描述自己的問題？

### 3. 現況
- 目前有沒有內容？什麼有效？
- 有哪些資源？（寫手、預算、時間）
- 能產出哪些格式？（文字、影片、音訊）

### 4. 競品態勢
- 主要競品是誰？
- 市場中有哪些內容缺口？


## Searchable vs Shareable

- **Searchable 內容**：SEO 導向，長尾關鍵字，持續帶流量（部落格、FAQ、教學）
- **Shareable 內容**：社群導向，情緒共鳴，短期爆發（迷因、故事、懶人包）
- 理想比例：70% searchable / 30% shareable（依產業調整）

## 產出格式

擬定內容策略時請提供：

### 1. 內容柱（Content Pillars）
- 3-5 個內容柱加上理由
- 每個內容柱的子題叢集
- 內容柱如何連結到產品

### 2. 優先主題
每個建議的內容提供：
- 主題／標題
- 屬於 Searchable、Shareable，或兩者兼具
- 內容類型（使用情境、主幹分支、思想領導等）
- 目標關鍵字與買家階段
- 為什麼選這個主題（客戶研究佐證）

### 3. 主題叢集圖
以視覺化或結構化方式呈現內容之間的關聯。


## 任務專屬提問

1. 從您最近 10 次客戶對話中，看到什麼樣的模式？
2. 業務電話中持續出現的問題是什麼？
3. 競品的內容在哪些地方做得不夠？
4. 客戶研究中有哪些獨特洞察還沒被別人分享？
5. 現有內容中哪一篇帶來最多轉換？為什麼？


## Proactive Triggers（主動觸發）

在 context 中察覺以下問題時，**不等使用者問**就提出：

- **沒有內容計畫** → 立刻提出 3 個內容柱的起手策略加 10 個種子主題，再問更多問題。
- **已有內容但流量低** → 指出 searchable 與 shareable 比例失衡；快速稽核現有標題與關鍵字意圖是否對齊。
- **使用者還沒找好關鍵字就開始寫** → 警告可能白費功夫；主動幫忙先鎖定正確關鍵字再動筆。
- **內容對象涵蓋過多受眾** → 指出 ICP 被稀釋；建議依人物誌或使用情境拆分內容柱。
- **競品在核心主題明顯輾壓** → 觸發缺口分析，指出競爭較低的快贏機會。


## Output Artifacts（產出物）

| 使用者要什麼 | Claude 給什麼 |
|---------------------|------------|
| 內容策略 | 3-5 個內容柱加理由、每柱子題叢集、產品—內容連結圖 |
| 主題發想 | 優先主題表（關鍵字、搜尋量、難度、買家階段、內容類型、分數） |
| 內容行事曆 | 週／月計畫：主題、格式、目標關鍵字、分發通路 |
| 競品分析 | 缺口表：競品涵蓋 vs 您的涵蓋，附機會評分 |
| 內容 Brief | 一頁式 brief：目標、受眾、關鍵字、大綱、CTA、內部連結、證據點 |


## Communication（溝通原則）

所有產出遵循結構化溝通標準：

- **先結論** —— 先給建議，再講理由
- **What + Why + How** —— 每個策略都同時包含這三項
- **行動有負責人與期限** —— 不寫「您或許可以考慮」
- **信心標註** —— 🟢 高信心 / 🟡 中等 / 🔴 假設待驗證

輸出格式預設：優先排序用表格、選項用條列、理由用散文。深度對應請求——簡單問題給簡短回答，不是丟一份策略文件。


## Do's and Don'ts

### Do
- 清楚勝過聰明——如果要在「有創意」和「看得懂」之間選，選看得懂
- 用客戶的語言寫文案——從評論、客服紀錄、訪談中抓他們實際用的字詞
- 一個段落只講一個論點，沿著頁面建立邏輯流
- CTA 用具體動詞 + 對方得到什麼：「開始免費試用」「下載完整指南」而非「送出」「了解更多」
- 台灣繁體中文文案注意繁簡用語差異（「按讚」非「點贊」、「貼文」非「帖子」）

### Don't
- 不要以「我們」或公司名開頭——先講對方的痛點或結果
- 不要用弱 CTA（Submit、Learn More、Sign Up、Click Here）——這些不傳達價值
- 不要列功能不附帶好處——每個功能後面要有「which means...」橋接到客戶得到什麼
- 不要忽略台灣繁簡用語差異——錯用「視頻」「博客」「點贊」會降低品牌信任感
- 不要一個選項就交差——高風險元素（headline、CTA）至少給 2-3 個選項

## 快速參考

### Landing Page 文案流程
1. 確認頁面目標（一個 CTA）+ 流量來源（廣告 / 自然 / Email）
2. 用 `query_knowledge` 載入品牌語氣、ICP、定位 context
3. 寫首屏 (Above the Fold)：主標 (Headline)（核心價值）→ 副標 (Subheadline)（展開）→ 主要行動呼籲 (Primary CTA)
4. 寫核心區段：社群背書 (Social Proof) → 問題／痛點 (Problem/Pain) → 解法／好處 (Solution/Benefits) → 運作方式 (How It Works) → 異議處理 (Objection Handling) → 最終行動呼籲 (Final CTA)
5. 每個 headline 和 CTA 附 2-3 個替代方案 + 選擇理由

### 內容策略規劃流程
1. 蒐集 context：公司做什麼、ICP 是誰、目標（流量 / 轉換 / 品牌）
2. 定義 3-5 個內容柱 + 各柱子的子題叢集
3. 分類每個主題：Searchable（SEO 長尾）vs Shareable（社群爆發）
4. 建立優先順序表：主題、關鍵字、搜尋量、難度、買家階段
5. 產出每週/月內容行事曆 + 分配負責人與截止日

### Headline 撰寫公式
1. 成果型：「{達成結果} 不用 {痛點}」
2. 分類型：「給 {受眾} 的 {產品類別}」
3. 消除型：「再也不用 {不愉快的事}」
4. 問題型：「{點出核心痛點的問題}？」
5. 數據型：「{具體數字} {具體結果}」
