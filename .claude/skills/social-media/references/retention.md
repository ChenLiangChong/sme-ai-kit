## churn-prevention


# Churn Prevention（流失預防）

本模組專注於留客與流失（Churn，流失率）預防。目標是透過聰明的流程設計、精準的挽留 Offer（Save Offer，挽留優惠）、系統化的付款恢復，降低兩種流失：自願流失（客戶決定離開）與非自願流失（因付款失敗而離開）。

流失是一個可以堵住的營收漏水口。對自願流失者 20% 的挽留率、對非自願流失者 30% 的恢復率，每月可回收 5-8% 失去的 MRR（月度經常性收入）。這會複利。

## 開工前

**先檢查既有的品牌／行銷脈絡：**
問問題之前，先用 `query_knowledge(category='brand')` 和 `query_knowledge(category='marketing')` 查詢既有脈絡。

收集以下脈絡（沒提供就問）：

### 1. 現況
- 今天有 cancel flow（取消流程）嗎，還是取消是即時／透過客服？
- 目前月流失率？（若知道，區分自願 vs. 非自願）
- 使用哪個金流？（Stripe、Braintree、Paddle 等）
- 目前有收集退出原因嗎？

### 2. 業務脈絡
- 商業模式：自助、業務協助，還是實體零售／服務？
- 定價與方案結構
- 平均合約長度與計費週期（月／年）
- 目前 MRR

### 3. 目標
- 主要問題：取消太多，還是付款失敗流失？
- 有挽留 Offer 預算嗎？（折扣、延長）
- 取消流程摩擦有限制嗎？（部分平台會懲罰暗黑模式）

## 本模組的運作模式

### Mode 1：建立 Cancel Flow
從零開始——沒有取消流程，或取消是即時的。我們設計從觸發到取消後的完整流程。

### Mode 2：優化既有流程
您有 Cancel Flow 但挽留率低或沒捕捉到好的退出資料。稽核現況、找出缺口、重建表現差的部分。

### Mode 3：設定 Dunning
付款失敗造成的非自願流失是您的優先問題。建立重試邏輯、通知序列、恢復 Email。


## Cancel Flow 設計

Cancel Flow 不是暗黑模式——是一段結構化對話。目標是理解為什麼離開，並提供真正有用的東西。如果對方仍想取消，就讓他走。

### 5 階段流程

```
[取消觸發 Cancel Trigger] → [退出調查 Exit Survey] → [動態挽留 Offer Dynamic Save Offer] → [確認 Confirmation] → [取消後 Post-Cancel]
```

**Stage 1 — Cancel Trigger（取消觸發）**
- 明確顯示取消選項（不要藏——暗黑模式燒信任）
- 他們點下取消的當下啟動流程——不要把他帶到死胡同表單
- 手機：觸控可用

**Stage 2 — Exit Survey（退出調查，1 題必填）**
- 只問一題：「您主要取消的原因是什麼？」
- 單選（最多 6-8 個原因）——開放文字可選，非必填
- 這個答案驅動 Save Offer——必須在給 Offer 前收集到

**Stage 3 — Dynamic Save Offer（動態挽留 Offer）**
- Offer 要配合原因（見下方 Exit Survey → Save Offer 對應表）
- 不要給通用折扣——這等於告訴客戶您定價是假的
- 一次只給一個 Offer。被拒就讓他取消。

**Stage 4 — Confirmation（確認）**
- 清楚摘要取消後會發生什麼（存取、資料、帳單）
- 明確的確認按鈕——"Yes, cancel my account"
- 不預先勾選、不用混淆語言

**Stage 5 — Post-Cancel（取消後）**
- 立即確認 Email：取消日期、資料保留政策、重啟連結
- 7 天後 re-engagement email：單一 CTA、無壓力、重啟連結
- 30 天後 win-back（贏回／再啟動，若合理）（產品更新或相關 Offer）


## 退出調查設計

調查是您最有價值的資料來源。設計成產生可用情報，不只是分類。

### 建議原因分類

| 原因 | 挽留 Offer | 訊號 |
|------|-----------|------|
| 太貴／價格 | 折扣或降級 | 價格敏感 |
| 用得不夠 | 使用提示 + 暫停選項 | 採用失敗 |
| 缺某個功能 | 分享 Roadmap + 暫代方案 | 產品缺口 |
| 轉用競品 | 競品比較 | 市場定位 |
| 專案結束／季節性 | 暫停選項 | 短暫需求 |
| 太複雜 | Onboarding 協助 + 真人支援 | UX 摩擦 |
| 只是測試／沒用過 | 不給 Offer——讓他走 | 契合度錯 |

**實作規則：**每個原因必須對應到一種挽留 Offer 類型。模糊對應 = 通用 Offer = 低挽留率。


## 挽留 Offer Playbook

Offer 配合原因。每種 Offer 都有適用與不適用時機。

| Offer 類型 | 何時用 | 何時不用 |
|-----------|-------|---------|
| **折扣**（1-3 個月） | 價格異議 | 採用或功能問題 |
| **暫停**（1-3 個月） | 季節性、專案結束、沒用 | 價格異議 |
| **降級** | 太貴、用量輕 | 功能異議 |
| **延長試用** | 還沒探索完整價值 | Power user 流失 |
| **功能解鎖** | 缺的功能在更高方案有 | 方案契合錯 |
| **真人支援** | 太複雜、卡住、受挫 | 價格異議（別浪費 CS 時間） |

**Offer 呈現規則：**
- 明確主標：「Before you go — [offer]」
- 量化價值：「省 $X」不是「拿折扣」
- 不要倒數計時除非真的到期
- 明確 CTA：「領取此 Offer」vs.「繼續取消」


## 非自願流失：Dunning 設定

付款失敗造成多數訂閱業務 20-40% 的總流失。大部分是可以救回來的。

### 恢復堆疊

**1. 智能重試邏輯**
不要立刻重試——失敗的卡通常在 3-7 天內恢復：
- Retry 1：失敗後第 3 天（多數恢復發生在此）
- Retry 2：Retry 1 後第 5 天
- Retry 3：Retry 2 後第 7 天
- Final：Retry 3 後第 3 天，然後取消

**2. Card Updater 服務**
- Stripe：Account Updater（自動，多數方案預設啟用）
- Braintree：Account Updater（需手動啟用）
- 下次扣款前更新過期／更換的卡——用它們

**3. Dunning Email 序列**

| 天數 (Day) | Email 主旨 | 口吻 | 行動呼籲 (CTA) |
|----|-------|-----|-----|
| 第 0 天 | "付款失敗" | 中性、陳述事實 | 更新卡片 |
| 第 3 天 | "需要您處理" | 輕微緊迫 | 更新卡片 |
| 第 7 天 | "帳戶有風險" | 較高緊迫 | 更新卡片 |
| 第 12 天 | "最後通知" | 緊急 | 更新卡片 + 支援連結 |
| 第 15 天 | "帳戶已暫停／取消" | 陳述事實 | 重啟 |

**Email 規則：**
- 主旨：具體優於模糊（"Your [Product] payment failed" 不是 "Action required"）
- 不羞辱、不責備。卡失敗很常見——以成人方式對待客戶。
- 每封信直接連到付款更新頁——不是儀表板


## 指標與基準

每週追蹤、每月回顧：

| 指標 | 公式 | 基準 |
|------|------|------|
| **挽留率（Save rate）** | 挽留的客戶 / 取消嘗試 | 10-15% 好、20%+ 優異 |
| **自願流失率** | 自願取消 / 總客戶 | 每月 <2% |
| **非自願流失率** | 付款失敗取消 / 總客戶 | 每月 <1% |
| **恢復率** | 恢復的付款 / 總失敗付款 | 25-35% 好 |
| **Win-back 率** | 重啟 / 取消後 90 天 | 5-10% |
| **退出調查完成率** | 完成調查 / 取消嘗試 | >80% |

**紅旗：**
- 挽留率 <5% → Offer 沒配對原因
- 退出調查完成率 <70% → 調查太長或設為選填
- 恢復率 <20% → 重試邏輯或 Email 需改善

用 churn impact 計算器建模每個指標改善的營收價值。

## Proactive Triggers（主動觸發）

不等使用者問就主動提出：

- **即時取消流程** → 營收正在立刻流失。任何摩擦都能省錢——標記為優先修正。
- **單一通用挽留 Offer** → 對所有人展示同一個折扣會壓低平均營收，並訓練客戶等促銷。把 Offer 對應到退出原因。
- **沒有 Dunning 序列** → 付款失敗後沒動作，等於 20-40% 的流失沒處理。立刻標記。
- **退出調查是選填** → <70% 完成率 = 資料不好。設為必填（一題、快）。
- **沒有取消後重啟 Email** → 7 天視窗是最高 Win-back 時機。錯過就把錢丟桌上。
- **月流失率 >5%** → 這個速度公司可能在萎縮。只靠留客修不了——標記需與產品／ICP 檢討並行。

## 實體/服務業留客策略

> 以下段落為台灣中小企業（餐飲、零售、服務業）的留客策略補充。訂閱制業務請參考上方的 cancel flow 和 dunning 設計。

### VIP 制度設計

| 等級 | 門檻（依業態調整） | 權益範例 |
|------|-----------------|---------|
| 一般會員 | 首次消費 | 生日禮、不定期優惠 |
| 銀卡 | 累計消費 NT$5,000+ | 95 折、優先預約 |
| 金卡 | 累計消費 NT$20,000+ | 9 折、專屬服務、新品搶先體驗 |
| VIP | 累計消費 NT$50,000+ | 88 折、專人服務、活動邀請 |

**設計原則：**
- 門檻要可達成 — 太高讓人放棄，太低沒感覺
- 權益要有感 — 折扣只是基本，體驗升級更有黏性
- 透過 LINE 通知等級升級 — 立即的正面回饋

### 會員卡/集點制度

- **實體集點卡** → 建議轉為 LINE 數位集點（減少遺失、自動追蹤）
- **集點兌換** — 設計合理的兌換門檻（消費 10 次 → 免費一次）
- **消費紀錄追蹤** — 透過 business-db 的 `log_interaction()` 和 `record_transaction()` 記錄

### LINE 會員經營

- **加好友即會員** — 降低入會門檻，掃碼即完成
- **標籤分群** — 依消費頻率、品類、金額自動分群
- **生命週期管理** — 新客歡迎 → 培養 → 活躍 → VIP → 沉睡喚醒
- **個人化推播** — 依消費紀錄推薦相關產品/服務

詳細 LINE 會員策略見 [line-marketing.md](line-marketing.md)

### 留客關鍵指標（實體/服務業）

| 指標 | 計算方式 | 健康基準 |
|------|---------|---------|
| 回購率 | 回購客數 / 總客數 | > 30% |
| 回購週期 | 平均回購間隔天數 | 視產業（餐飲 7-14 天、零售 30-60 天） |
| 客單價變化 | 新客 vs 老客平均消費 | 老客應 > 新客 20%+ |
| LINE 好友留存 | 未封鎖好友 / 總好友 | > 70% |
| VIP 貢獻比 | VIP 營收 / 總營收 | 20-30%（頭部客戶貢獻高） |


## Do's and Don'ts

### Do
- Cancel flow 先收集退出原因再給 save offer——原因決定 offer 類型
- 每個退出原因對應一個具體 offer（價格 → 折扣/降級、不常用 → 暫停、缺功能 → roadmap）
- Dunning 郵件要直接連到付款更新頁——不要連到 dashboard 讓用戶自己找
- 退出調查設為必填（一題選擇題即可）——完成率低於 70% 代表調查太長

### Don't
- 不要用暗黑模式隱藏取消按鈕——會燒掉信任，部分平台還會處罰
- 不要給所有人同一個通用折扣——這等於告訴用戶你的定價是假的
- 不要在 dunning 郵件中使用羞辱語氣——信用卡失敗很常見，以成人方式對待客戶
- 不要跳過 exit survey——沒有退出原因資料就無法改善產品和流程

## 快速參考

### Cancel Flow 建置
1. 取消觸發 → 退出調查（1 題必填選擇題）→ 動態 Save Offer → 確認取消 → 取消後處理
2. 每個原因對應一個 offer（見 Exit Survey → Save Offer Mapping 表）
3. 取消後立即寄確認信 → 7 天後寄 re-engagement → 30 天後考慮 win-back

### Dunning（付款失敗恢復）
1. 啟用 Card Updater（Stripe 預設開啟）
2. 重試排程：失敗後第 3 天 → 第 8 天 → 第 15 天 → 第 18 天取消
3. 每次重試搭配一封 dunning email，語氣從中性到緊急遞進，CTA 一律連到付款更新頁

### 實體/服務業留客（台灣）
1. 設計 VIP 等級制度（門檻要可達成、權益要有感）
2. 集點卡轉為 LINE 數位集點 → 自動追蹤消費紀錄
3. LINE 會員分群推播：依消費頻率和品類個人化推薦
