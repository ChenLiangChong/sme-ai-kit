# 文案編修

本模組負責行銷與轉換型文案的編修，目標是透過聚焦式的編輯掃描，在保留核心訊息的前提下系統化地強化既有文案。

## 核心理念

**先檢查既有的品牌／行銷 context：**
在提問之前，先用 `query_knowledge(category='brand')` 和 `query_knowledge(category='marketing')` 檢查既有 context。

好的文案編修不是重寫，而是強化。每一次掃描只聚焦一個維度，抓出「一次想改所有問題」會漏掉的東西。

**核心原則：**
- 不改核心訊息，專注於強化它
- 多次聚焦掃描，勝過一次不聚焦的審閱
- 每次編輯都要有明確理由
- 保留作者的聲音，同時提升清楚度


## 七掃編輯框架（The Seven Sweeps Framework）

依序以七次掃描編修文案，每次聚焦一個維度。每完成一次掃描，回頭檢查前面的掃描是否被破壞。

### Sweep 1：Clarity（清楚度）

**聚焦：** 讀者看得懂您在講什麼嗎？

**要檢查什麼：**
- 令人困惑的句型
- 不清楚的代名詞指涉
- 行話或內行語言
- 模稜兩可的陳述
- 缺漏的 context

**常見的清楚度殺手：**
- 一句話想講太多事
- 抽象語言取代具體
- 假設讀者知道他們其實不知道的事
- 把重點埋在修飾語裡

**流程：**
1. 快速讀一遍，把不清楚的部分標出來
2. 先別改——只先記下問題點
3. 標完問題後，再提出具體修改建議
4. 確認修改保留原意

**掃描後：** 確認「單一主軸原則 (Rule of One)」（每段只講一個主要觀點）與「讀者為本原則 (You Rule)」（文案以讀者為對象）仍然完整。


### Sweep 2：Voice and Tone（語氣與口吻）

**聚焦：** 文案聽起來一致嗎？

**要檢查什麼：**
- 在正式與口語之間的切換
- 品牌人格不一致
- 令人錯愕的情緒轉折
- 用字選擇與品牌不符

**常見語氣問題：**
- 一開始口語、後來變官方
- 「we」和「the company」混用
- 有些地方幽默、有些地方嚴肅（非刻意）
- 技術語言隨機出現

**流程：**
1. 朗讀出聲，聽出不一致
2. 標出口吻突然跳動的地方
3. 建議可以把轉折順起來的修改
4. 確保人格從頭貫穿到尾

**掃描後：** 回到 Clarity Sweep，確認語氣修改沒有帶來新的混淆。


### Sweep 3：So What（所以呢）

**聚焦：** 每個主張都回答了「我為什麼要在乎？」嗎？

**要檢查什麼：**
- 只有功能沒有好處
- 主張沒有後續結果
- 陳述與讀者的生活脫鉤
- 缺少「which means...」橋接

**So What 測試：**
對每個陳述問一次「OK，所以呢？」。如果文案沒有用更深一層的好處回答，就需要再寫一次。

❌ 「Our platform uses AI-powered analytics」
*所以呢？*
✅ 「Our AI-powered analytics surface insights you'd miss manually—so you can make better decisions in half the time」

**常見 So What 失誤：**
- 功能清單沒有好處連結
- 聽起來很厲害但沒落地的主張
- 技術能力沒有帶出成果
- 公司成就對讀者沒幫助

**流程：**
1. 讀每個主張時，直接問「so what?」
2. 把沒有回答的主張標出來
3. 加上好處橋接或更深一層的意義
4. 確保好處連結到讀者真正的渴望

**掃描後：** 回到 Voice and Tone，再回到 Clarity。


### Sweep 4：Prove It（拿出證據）

**聚焦：** 每個主張都有證據支持嗎？

**要檢查什麼：**
- 沒有佐證的主張
- 缺少社群背書
- 沒有後盾的斷言
- 「Best」或「Leading」沒有證據

**可找的證據類型：**
- 有姓名與細節的客戶見證
- 案例研究引用
- 統計與數據
- 第三方認證
- 保證與反向風險保證
- 客戶 Logo
- 評價分數

**常見證據缺口：**
- 「Trusted by thousands」（哪個 thousands？）
- 「Industry-leading」（誰說的？）
- 「Customers love us」（拿出他們說的話）
- 成果主張沒有具體數字

**流程：**
1. 找出每個需要證據的主張
2. 檢查附近有沒有證據
3. 把沒有支持的斷言標示出來
4. 建議補證據或把主張弱化

**掃描後：** 回到 So What、Voice and Tone、再回到 Clarity。


### Sweep 5：Specificity（具體度）

**聚焦：** 文案夠具體、夠有說服力嗎？

**要檢查什麼：**
- 含糊用語（「improve」「enhance」「optimize」）
- 誰都能用的通用陳述
- 聽起來像掰的整數
- 缺少能讓事情變真實的細節

**具體度升級：**

| 含糊 | 具體 |
|-------|----------|
| Save time | Save 4 hours every week |
| Many customers | 2,847 teams |
| Fast results | Results in 14 days |
| Improve your workflow | Cut your reporting time in half |
| Great support | Response within 2 hours |

**常見具體度問題：**
- 形容詞做了名詞應該做的工作
- 好處沒有量化
- 成果沒有時間框
- 主張沒有具體範例

**流程：**
1. 把含糊字詞與片語標出來
2. 問「這能不能更具體？」
3. 加入數字、時間框或範例
4. 刪除無法變具體的內容（多半是填充物）

**掃描後：** 回到 Prove It、So What、Voice and Tone、再回到 Clarity。


### Sweep 6：Heightened Emotion（情緒強度）

**聚焦：** 文案能讓讀者感受到什麼嗎？

**要檢查什麼：**
- 平淡、只有資訊的語言
- 缺少情緒觸發
- 痛點被提及但沒有被感受到
- 渴望被陳述但沒有被召喚

**可考慮的情緒面向：**
- 現況的痛苦
- 對替代方案的挫敗
- 害怕錯過
- 對蛻變的渴望
- 做出聰明選擇的自豪
- 解決問題後的如釋重負

**強化情緒的技巧：**
- 生動描繪「before」狀態
- 使用感官語言
- 說微故事
- 引用共同經驗
- 問能引發反思的問句

**流程：**
1. 讀出情緒影響——它有感動您嗎？
2. 找出應該引發共鳴卻很平淡的段落
3. 在不失真的前提下加入情緒肌理
4. 確保情緒服務訊息（不是操弄）

**掃描後：** 回到 Specificity、Prove It、So What、Voice and Tone、再回到 Clarity。


### Sweep 7：Zero Risk（零風險）

**聚焦：** 有沒有把行動的每一道障礙都移除？

**要檢查什麼：**
- CTA 附近的摩擦
- 未被處理的異議
- 缺少信任訊號
- 不清楚的下一步
- 隱藏成本或意外

**可找的風險消除器：**
- 退款保證
- 免費試用
- 「No credit card required」
- 「Cancel anytime」
- CTA 附近的社群背書
- 清楚說明下一步會發生什麼
- 隱私保證

**常見風險問題：**
- CTA 在還沒贏得信任時就要求承諾
- 異議被提起但沒被處理
- 小字引起疑慮
- 模糊的「Contact us」沒有明確下一步

**流程：**
1. 聚焦在 CTA 附近的段落
2. 列出對方可能猶豫的每個理由
3. 檢查文案是否對每個顧慮都有回應
4. 依需要補上反向風險保證或信任訊號

**掃描後：** 把前面所有掃描再跑一次：Heightened Emotion、Specificity、Prove It、So What、Voice and Tone、Clarity。


## 快速檢查式編修

在不需要跑完整七掃流程時，使用這些加速檢查。

### 字詞層級檢查

**砍掉這些字：**
- Very、really、extremely、incredibly（弱強化詞）
- Just、actually、basically（填充）
- In order to（用 「to」）
- That（常常多餘）
- Things、stuff（含糊）

**替換這些：**

| 弱 | 強 |
|------|--------|
| Utilize | Use |
| Implement | Set up |
| Leverage | Use |
| Facilitate | Help |
| Innovative | New |
| Robust | Strong |
| Seamless | Smooth |
| Cutting-edge | New/Modern |

**留意：**
- 副詞（通常多餘）
- 被動語態（改主動）
- 名詞化（動詞 → 名詞：「make a decision」 → 「decide」）

### 句子層級檢查

- 一句一個觀點
- 句長多樣（長短交替）
- 重要資訊前置
- 一句最多 3 個連接詞
- 不超過 25 字（通常）

### 段落層級檢查

- 一段一個主題
- 短段落（網頁用 2-4 句）
- 段首句要強
- 段與段之間邏輯順暢
- 留白以利掃讀

## 常見文案問題與對策

### 問題：功能牆
**症狀：** 只列產品做什麼，沒講為什麼重要
**對策：** 每個功能後加「which means...」橋接到好處

### 問題：官僚語
**症狀：** 「Leverage synergies to optimize outcomes」
**對策：** 問「人會怎麼說這句？」然後用那些字

### 問題：弱開頭
**症狀：** 以公司歷史或含糊陳述開頭
**對策：** 以讀者的問題或期望成果領頭

### 問題：CTA 被埋
**症狀：** 要求的行動埋在太多鋪陳之後，或不清楚
**對策：** 讓 CTA 明顯、提前、重複

### 問題：沒有證據
**症狀：** 「客戶喜愛我們」沒有佐證
**對策：** 加具體客戶見證、數字或案例引用

### 問題：通用主張
**症狀：** 「We help businesses grow」
**對策：** 具體化：誰、怎麼做、做多少

### 問題：多受眾混雜
**症狀：** 試圖對所有人講話，反而沒人有共鳴
**對策：** 挑一個受眾，直接對他寫

### 問題：功能過載
**症狀：** 把每個功能都列出來，壓垮讀者
**對策：** 聚焦 3-5 個對該受眾最重要的好處


## 協作式編修流程

協作編修時：

1. **跑完一次掃描後呈現發現** —— 指出發現什麼、為什麼是問題
2. **建議具體修改** —— 不只指問題，要提解法
3. **請求更新後的文案** —— 讓作者做最終決定
4. **驗證前面的掃描** —— 每輪修改後，回頭檢查前面的掃描
5. **重複直到乾淨** —— 直到一次完整掃描找不到新問題

這個迭代流程能確保每次修改不會產生新問題，同時尊重作者對文案的歸屬。

## Proactive Triggers（主動觸發）

在 context 中察覺以下問題時，**不等使用者問**就提出：

- **文案被送來編修但沒有說明目標** → 在開始任何掃描前，先問目標行動與受眾；沒有這個 context 的編輯必然是對錯頻道的回饋。
- **偵測到多處語氣跳動** → 立刻標示 Sweep 2 失敗；指出口吻斷裂的具體行，先提修法再繼續。
- **功能與好處比例是 2:1 或更高** → 在審閱早期就拉響「So What」警報；這是最常見的轉換殺手。
- **誇大詞沒有證據**（「best」「leading」「most trusted」）→ 在 Sweep 4 把每一處標出來，並要求證據或提出較軟的替代語言。
- **CTA 含糊或被埋** → 在 Sweep 7 於任何其他回饋之前先提出來——這是影響最大的修法。

## Communication（溝通原則）

所有產出遵循結構化溝通標準：

- **先結論** —— 在深入問題之前，先講整體文案健康度
- **What + Why + How** —— 每個問題都要有：哪裡錯、為什麼傷轉換、怎麼修
- **編輯要有理由** —— 不在沒說明原則的情況下改字
- **信心標註** —— 🟢 明顯改善 / 🟡 判斷題 / 🔴 需要作者輸入

逐掃分批交付。不要一次倒出所有問題。按轉換影響排序，不是按寫作偏好。


## Do's and Don'ts

### Do
- 七掃依序執行——每次 sweep 只聚焦一個維度（Clarity → Voice → So What → Prove It → Specificity → Emotion → Zero Risk）
- 每次修改都要有理由——不要只改字，要解釋為什麼改、對轉換有什麼影響
- 保留作者的聲音——editing 的目標是強化原有訊息，不是改寫成你的風格
- 每完成一個 sweep 後回頭檢查前面的 sweep 是否被破壞
- 開始前先確認目標行動和受眾——沒有 context 的編輯等於盲修

### Don't
- 不要沒有目標就開始編輯——先問「這份文案要達成什麼目標？讀者是誰？」再動手
- 不要一次改所有東西——用多次聚焦 pass，每次只改一個維度；一次改全部必然漏東西
- 不要把 copy-editing 當 copywriting——編輯是強化現有文案，不是從頭重寫；需要重寫請用 copywriting skill
- 不要改了字不說理由——每個修改都要附上原則依據（clarity / voice / so-what / prove-it / specificity / emotion / zero-risk）

## 快速參考

### 完整七掃編輯流程
1. **Clarity**：每句話看得懂嗎？有沒有術語、模糊代名詞、一句話講太多事？
2. **Voice & Tone**：語氣一致嗎？有沒有突然從口語變正式、或品牌人格斷裂？
3. **So What**：每個主張都回答了「所以呢？」嗎？功能有接到好處嗎？
4. **Prove It**：每個主張有證據嗎？「業界領先」「客戶喜愛」有具體佐證嗎？
5. **Specificity**：有沒有模糊用語（improve、enhance、optimize）可以換成具體數字？
6. **Heightened Emotion**：文案有讓人「感受到什麼」嗎？痛點有畫面感嗎？
7. **Zero Risk**：CTA 附近有沒有未解決的顧慮？有沒有信任信號（保證、試用、社群背書）？

### Quick-Pass 快速編輯（不需要完整七掃時）
1. 砍弱詞：very / really / just / actually / basically / in order to
2. 換弱動詞：utilize → use / implement → set up / leverage → use / facilitate → help
3. 句子層級：一句一個觀點、長短交替、重點前置、不超過 25 字
4. 段落層級：一段一個主題、2-4 句（網頁）、段間邏輯通順

### 常見問題速查
1. **功能牆**（只列功能沒好處）→ 每個功能加「which means...」
2. **弱開頭**（公司歷史或空泛陳述）→ 改成讀者的痛點或期望結果開頭
3. **CTA 被埋**（太深或不明確）→ CTA 提前、重複、動詞 + 對方得到什麼
4. **沒有證據**（「客戶喜愛」無佐證）→ 加具名推薦、數字、案例
