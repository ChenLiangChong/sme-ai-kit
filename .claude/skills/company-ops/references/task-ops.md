# 任務管理專業指南（律所版）

> **適用**：律師事務所所內案件任務與行政待辦。所內成員＝律師 / 助理 / 行政；交辦人＝所長 / 主持律師。

---

## ⚠️ 硬邊界：task.due_date ≠ 法定時限（最重要、優先讀）

**`task.due_date` 是內部行政提醒日，絕對不等於任何法定時限。**

- 上訴 / 抗告 / 答辯 / 準備書狀 / 補正 / 訴願 / 異議 / 消滅時效等**一切受法律規範的期間**，一律走 **legal-admin** 技能包的 `create_deadline` 確定性計算、附 `statutory_basis` 法條依據，**不可建成 task**。
- task 模組沒有任何時限計算能力：它不查法條、不算送達 / 在途、不順延末日、不附 `statutory_basis`。把法定期間塞進 task 的 due_date ＝ 把「執業過失風險」藏進一個只會做提醒的欄位。
- **本檔所有範例的 due_date 都不得出現任何法定期間天數**（不寫「上訴 N 天」「補正 N 天」這類數字）。律所的法定期間天數一律是 legal-admin 引擎查表 / 計算的事，文件不心算、不舉例。
- task 的 `due_date`（內部行政提醒）與 leave 的請假天數，**與法定時限天數零關係**，三者不可互相換算或類比。

**判斷準則**：這件事「算錯日期會構成執業過失 / 喪失程序權利」嗎？
- 會 → 它是法定時限 → 交 legal-admin `create_deadline`（見 `legal-admin/references/deadline-intake.md`），**不是 task**。
- 不會（純內部進度 / 行政提醒，如「撰狀草稿初版交主辦」「開庭前一天備齊卷宗」）→ 才用 task，且 due_date 是律所自己排的軟性提醒、可由所長調整。

> 即使律師口頭說「幫我記一下這個上訴期限」，正解仍是進 legal-admin 走確定性計算（HITL 一鍵確認才入庫），**不是**在這裡建一筆 task。task 只承接「不受法律規範的內部行政待辦」。

## 觸發情境

「建立任務」「指派給助理甲」「我的待辦」「進度如何」「做完了」「任務列表」
「這週要做什麼」「哪些逾期了」「幫我排優先」「會議紀錄轉任務」

> 注意：若觸發語句帶法定期間語意（上訴 / 抗告 / 答辯 / 補正 / 時效 / 開庭通知要算到期日），那不是 task 情境，轉 legal-admin。本檔只處理「所內行政待辦 / 案件作業進度」。

## 一、任務建立流程

### 從對話中提取任務

1. 從對話中提取：標題、描述、指派對象、優先級、截止日期（**內部行政提醒日、非法定時限**）、category（分類）
2. 如果有指派對象，先 `lookup_employee` 確認人存在
3. 呼叫 `create_task`
4. 指派對象有綁 LINE → agent 主動用 `reply` 通知：「你有一項新任務：{標題}，截止 {日期}」（系統不自動推播給被指派人；通知是否送得到、由當前 floor 能否觸及該 LINE 脈絡決定，見 line-comms 執行模型）

> `lookup_employee` 屬 HR 工具，受 floor gate——floored 非全權限層呼不到屬正常、需交有該工具的層 / 全權限層執行（見 CLAUDE.md〈部門安全層（floor）與兩道牆〉、line-comms 第六節執行模型）。受限層在這種情況下：可直接 `create_task` 帶 `assignee` 字串（跳過確認），或把「確認成員 + 通知」交回全權限層。
>
> （個人律所通常全權限單人、不設 `SME_FLOOR`，floor 機制 inert；上述 caveat 在多人所或日後升級分層時才生效。）

### category 分類

`create_task` 支援的 category 值（擇律所相關者用）：

| category | 用途 |
|----------|------|
| `general` | 一般任務（預設） |
| `admin` | 行政庶務（歸檔、繳費、用印、寄送、印製） |
| `meeting` | 會議 / 所務會議行動項目 |

> 底層 `category` 為自由字串、預設 `general`；上表只列律所常用值。**任何案件「法定期間」都不該用 task category 承接**（見檔首硬邊界），category 只是替內部行政待辦分群、不賦予時限語意。

缺少資訊時主動問，不要猜：
- 沒說指派給誰 → 「這個任務要指派給誰？」
- 沒說截止日 → 「有內部截止日嗎？（提醒：這是行政提醒日，不是法定期間）」
- 沒說優先級 → 預設 normal，不用問

### 任務分解框架（大專案拆成小任務）

當使用者描述的是一個大目標（例：「準備這件言詞辯論」「年度卷宗大歸檔」「事務所搬遷」），主動進行拆解：

**拆解原則 — SMART 任務標準：**
- **S** 明確（Specific）：做什麼很清楚，不模糊
- **M** 可衡量（Measurable）：完成時有明確產出物
- **A** 可指派（Assignable）：能交給一個人負責
- **R** 合理（Realistic）：1-5 個工作天能完成
- **T** 有時限（Time-bound）：有**內部行政截止日**（非法定時限）

**拆解步驟：**
1. 確認最終目標和內部期限
2. 列出所有必要步驟（大約 5-15 個）
3. 識別先後依賴關係（哪些要先做完才能做下一步）
4. 為每個步驟估時間
5. 逐一建立任務，在描述中註明依賴關係

**拆解範例 —「開庭前準備（言詞辯論庭）」**（內部作業排程、各步驟的「天數」是事務所自排的工作量估計，**與法定期間無關**）：

| # | 任務 | 負責人 | 內部估時 | 依賴 |
|---|------|--------|---------|------|
| 1 | 調卷、整理本案卷宗目錄 | 助理 | 2天 | — |
| 2 | 影印爭點相關證物、製作證物清單 | 行政 | 1天 | #1 |
| 3 | 整理對造書狀、列出待反駁爭點 | 主辦律師 | 3天 | #1 |
| 4 | 撰擬言詞辯論要旨草稿初版 | 主辦律師 | 3天 | #3 |
| 5 | 草稿交主持律師覆閱、回修 | 主持律師 | 2天 | #4 |
| 6 | 聯絡當事人確認到庭、提醒攜帶文件 | 助理 | 1天 | #5 |
| 7 | 開庭前一日備齊卷宗、預演動線 | 主辦律師 | 1天 | #6 |

> 注意：上表**完全不涉及**「言詞辯論期日」「上訴期間」等法定日期。期日 / 法定期間請於 legal-admin 建 `create_deadline`（確定性計算附 `statutory_basis`），task 只承接「為了趕上那個日子、所內要先做完的行政與作業步驟」。兩者分流：法定期間在 legal-admin、內部備齊作業在 task。

當使用者只說「幫我建個任務：準備這件開庭」→ 回覆：
「這是個多步驟案件作業，我建議拆成以下內部步驟……你覺得這樣合理嗎？確認後我逐一建立。（開庭期日與任何法定期間我會另外在案件時限那邊處理，不會放進這些待辦的截止日。）」

## 二、優先級判斷框架

### 艾森豪矩陣（律所版）

當使用者問「幫我排優先」或有超過 5 個待辦需要排序時：

|  | 急迫-高 | 急迫-低 |
|---|---------|---------|
| **重要-高** | ① 立刻做（今天處理） | ② 排時間做（本週完成） |
| **重要-低** | ③ 交給別人做（委派） | ④ 考慮不做（觀察清單） |

**判斷「重要性」的準則：**
- 高：影響案件勝敗 / 當事人權益、主持律師親自交辦、攸關所譽
- 低：內部流程優化、nice-to-have、學習性質

**判斷「急迫性」的準則：**
- 高：今明天到期、當事人在等回覆、有外部行政時點（例如約定要在某日前把草稿送主辦律師覆閱）
- 低：截止日在一週以上、沒有外部壓力

> 此處「急迫性」指**內部行政急迫**。真正的程序急迫（迫近的上訴 / 抗告 / 補正末日）不靠這張矩陣判斷——那是 legal-admin 即將到期時限的事（`list_upcoming_deadlines`），且永遠優先於任何 task。

**優先級調整準則（agent 判斷後改、系統不自動升級）：**
- 內部截止日 ≤ 2 天且重要 → agent 建議 / 改 `update_task(task_id, priority='urgent')`
- 已逾期 → agent 視為 `urgent` 處理（DB 不會自己改 priority）
- 無截止日且 low → 啟動流程或週回顧時由 agent 判斷是否提醒所長「這個任務還需要嗎？」

> tasks 模組無排程 / 自動標記：以上是 agent 在處理任務時主動套用的判斷、不是系統定時跑的邏輯。

### 每日任務建議排序

當使用者問「今天要做什麼」，按以下順序排列：
1. 已逾期的任務（紅色警示）
2. 今天到期的任務
3. urgent 且 in_progress
4. urgent 且 pending
5. normal 且今天到期
6. 其餘按截止日排序

> 這只是**所內行政待辦**的排序。每日工作的全貌（含當日 / 迫近的法定時限、開庭、諮詢預約）由 legal-admin 的每日彙整負責（見 `legal-admin/references/daily-digest.md`）；法定時限永遠排在行政待辦之前。

## 三、狀態工作流

### 標準狀態流轉

DB 支援的狀態值：`pending` | `in_progress` | `done` | `cancelled`

```
pending → in_progress → done
  │
  └→ cancelled
```

### 卡住（blocked）處理

DB 沒有 `blocked` 狀態。當任務被阻擋時（例：等當事人補件、等對造提出書狀、等卷宗調閱）：
1. 狀態維持 `in_progress`（或視情況退回 `pending`）
2. 在描述前方加註：`[BLOCKED] 原因：{原因}`
3. agent 可主動通知建立者（`reply`）；要通知主持律師走 line-comms 執行模型，系統不會自動上報阻擋
4. 阻擋解除後移除 `[BLOCKED]` 標記
5. 查詢所有被擋任務：`search_tasks(query='[BLOCKED]')`

> 提醒：若「卡住」的原因是某個**法定期間迫近**（例：補正末日將至但當事人遲未補件），那不只是 task 卡住——對應的 `create_deadline` 時限本身需在 legal-admin 追蹤 / 提醒，不能只在 task 描述加 `[BLOCKED]` 了事。

### 狀態操作規則

| 動作 | 觸發語句 | 操作 | 後續（agent 判斷、非系統自動） |
|------|---------|------|------|
| 開工 | 「開始做了」「我在處理#3」 | `update_task(task_id=3, status='in_progress')` | 若建立者有綁 LINE、agent 可主動 `reply` 告知（系統不自動通知） |
| 完成 | 「做完了」「#3 完成了」 | `update_task(task_id=3, status='done')` | 同上、agent 視情況通知建立者 |
| 卡住 | 「等當事人補件」「被擋住了」 | 維持 `in_progress`，描述加註 `[BLOCKED] {原因}` | agent 可 `reply` 通知建立者；主持律師通知走 line-comms 執行模型，非系統自動上報 |
| 取消 | 「取消#5」「不做了」 | 先確認 → `update_task(task_id=5, status='cancelled')` | agent 視情況通知相關人 |

> `update_task` 第一參數是 `task_id`（不是 `id`）。tasks 模組只有 CRUD / search、本身不發通知；上表「後續」是 agent 接著做的動作，不是系統自動行為。

## 四、到期提醒

系統**不會**定時掃任務 due_date 主動推播。任務到期提醒靠兩個入口、都是 agent 當下判斷後處理：
- **啟動流程**：`get_context_summary` 啟動 readout 會帶出待處理任務（含截止日），agent 看到逾期 / 即將到期的、在當次對話裡提醒所長
- **agent 主動 follow-up**：發現逾期或將到期、可建一筆提醒 task 或直接 `reply` 通知負責人

下表是 agent 判斷「該不該提醒、提醒誰」的準則（**非系統自動排程的通知時程**），且僅適用於**內部行政截止日**：

| 時間點 | agent 處理建議 | 提醒對象（agent 自行 reply / 建 task） |
|--------|------|---------|
| 到期前 2 天 | 提醒 | 負責人 |
| 到期前 1 天 | 強調提醒 | 負責人 |
| 到期當天 | 警告語氣 | 負責人 |
| 逾期 1 天 | 逾期提醒 | 負責人 + 建立者 |
| 逾期 3 天 | 建議升級處理 | 負責人 + 建立者 + 主持律師（通知走 line-comms 執行模型） |
| 逾期 7 天 | 建議檢討、取消或重新排期 | 主持律師 |

> 上表的「天數」是**對內部行政截止日的提醒節奏**、與任何法定期間無關。法定時限的提醒（迫近 / 逾期、回復原狀）由 legal-admin 負責，且機制不同：那是引擎算出的確定日期 + 哨兵 / 每日彙整推播，不是這張 agent 行為表。
>
> 任務逾期**不**走 escalation 自動上報（escalation 觸發限於審核 / 員工權限 / 上報機制定義的高風險動作，見 CLAUDE.md〈上報（escalation）機制〉），任務的提醒與升級都是 agent 行為。

## 五、每週回顧

當使用者說「週回顧」「這週做了什麼」，執行：

**資料收集：**
- `list_tasks(status='done')` → 本週完成
- `list_tasks(status='pending')` → 待辦
- `list_tasks(status='in_progress')` → 進行中

**產出格式：**
```
第 XX 週回顧（MM/DD - MM/DD）

本週完成：X 項
進行中：X 項
待辦：X 項
逾期：X 項

完成率：XX%
準時率：XX%
```

**健康指標：**

| 指標 | 健康 | 警告 | 危險 |
|------|------|------|------|
| 週完成率 | ≥ 80% | 60-80% | < 60% |
| 準時率 | ≥ 70% | 50-70% | < 50% |
| 逾期任務數 | 0-2 | 3-5 | > 5 |

> 此回顧只盤點**所內行政待辦**的健康度。它**不取代**對案件法定時限的盤點——後者請看 legal-admin 即將到期時限清單（`list_upcoming_deadlines`），且「準時率」這類軟性指標**絕不**套用在法定期間上（法定期間沒有「達標率」，只有「準時遞交」與「逾期＝喪失權利」二態）。

## 六、委派原則（多人所團隊）

### 工作量平衡檢查

指派任務前，檢查該成員目前的工作量：

| 進行中+待辦數量 | 建議 |
|-----------------|------|
| 0-3 項 | 正常指派 |
| 4-6 項 | 提醒所長：「{成員}目前有 X 項任務」 |
| 7+ 項 | 建議重新分配 |

### 委派品質檢查

建立任務時，確認描述包含：
- [ ] 期望的產出物（例：證物清單、草稿初版、調卷回報）
- [ ] 內部截止時間
- [ ] 完成標準
- [ ] 遇到問題找誰

## 七、會議 → 行動項目

當使用者說「整理所務會議紀錄」「剛開完會」：

1. 抽取所有「誰要做什麼、什麼時候完成」
2. 只有明確有負責人+動作的才建任務
3. 排除純資訊分享、待討論、FYI
4. 逐一建立任務；建完後 agent 可主動逐一 `reply` 通知被指派的人（系統不自動通知，且 `lookup_employee` 確認身份受 floor gate、見上方第一節 caveat）

| 會議中的說法 | 轉換 |
|------------|------|
| 「助理甲你去調這件的卷」 | 任務：調卷，指派助理甲 |
| 「下週前把這件的草稿初版交出來」 | 任務：完成草稿初版，截止下週五（內部行政日） |
| 「這個之後再說」 | 不建任務 |
| 「大家回去想想」 | 不建任務（無明確負責人） |
| 「這件的上訴期限要記得算」 | **不建 task** → 轉 legal-admin 建 `create_deadline`（確定性計算附 `statutory_basis`） |

> 最後一列是律所最常見的誤分流：開會口頭提到的「期限」一旦是法定期間，絕不轉成 task 的 due_date——它必須進 legal-admin 走 HITL 一鍵確認 + 確定性計算。

## 八、常見陷阱

| 陷阱 | 處理 |
|------|------|
| **把法定期間建成 task** | **最嚴重**：上訴 / 抗告 / 補正 / 時效等一律轉 legal-admin `create_deadline`，不放進 task due_date（見檔首硬邊界） |
| 口頭交辦不記錄 | 提醒：「要不要建個任務追蹤？」 |
| 任務描述太模糊 | 主動問：「完成標準是什麼？」 |
| 全部都很急 | 套用優先級矩陣強制排序 |
| 一人扛太多 | 工作量平衡檢查 |
| 只建不關 | 週回顧時清理 |
| pending 超 14 天 | 啟動 / 週回顧時 agent 提醒：「這個任務是不是可以取消？」 |
| in_progress 超 7 天 | 啟動 / 週回顧時 agent 追蹤：「進度如何？需要幫忙嗎？」 |

> 上述「超 N 天」靠 agent 在啟動 readout 或週回顧時掃 `list_tasks` 結果判斷，系統不會定時自動跑或推播。（這裡的「N 天」是行政待辦的停滯天數，與法定期間無關。）

## 九、任務附件

任務相關的檔案（草稿底稿、證物影本掃描、參考資料等）：
`add_attachment(target_type='task', target_id=3, file_path='data/media/tasks/3/檔名', description='描述')`（`target_type` / `target_id` / `file_path` 為必填，`target_id` 帶實際任務 ID）

> 判決書 / 裁定 / 開庭通知等**要據以計算法定時限的法律文書**，不只是 task 附件——應走 legal-admin 收檔 → 抽取 → HITL 確認 → `create_deadline` 流程（見 `legal-admin/references/deadline-intake.md`）。task 附件只放「不觸發時限計算」的內部作業檔。

## Do's and Don'ts

### Do
- 缺少資訊主動問（指派對象、內部截止日），不要猜
- 每次狀態變更後 agent 接著 `log_interaction`（tool 不會自動寫）
- 啟動 / 週回顧時掃 `list_tasks`，對 `in_progress` 超 7 天的追蹤：「進度如何？需要幫忙嗎？」
- 啟動 / 週回顧時對 `pending` 超 14 天的提醒：「這個任務是不是可以取消？」
- 指派前 `list_tasks(assignee=...)` 檢查成員工作量（4-6 項提醒所長，7+ 項建議重新分配）
- 一發現某「待辦」其實是法定期間 → 立即停手、轉 legal-admin，不要建成 task

### Don't
- **不要把任何法定期間（上訴 / 抗告 / 答辯 / 補正 / 時效 / 開庭通知要算的日期）建成 task**，那是 legal-admin `create_deadline` 的事
- 不要在 task 範例或描述裡心算 / 寫死法定期間天數
- 不要猜 deadline（沒說就問「有內部截止日嗎？」）
- 不要用子任務（保持扁平）
- 不要逐一回覆批量建立的結果（統一報告）
- 不要把模糊的「之後再說」「大家回去想想」建成任務
- 不要在 LINE 通知中超過 3 行

## 快速參考

### 建立任務
1. `lookup_employee(name_or_line_id='助理甲')` — 確認人存在（HR 工具，受 floor gate；floored 非全權限層呼不到屬正常，見第一節 caveat）
2. `create_task(title='影印爭點證物、製作證物清單', assignee='助理甲', priority='normal', due_date='2026-06-12', category='admin', created_by='主持律師')`（`due_date` 為**內部行政提醒日**、非法定時限）
3. agent 主動 `reply(channel_id=成員所在 OA 的 channel_id, chat_id=助理甲的LINE_user_id, text='你有一項新任務：影印爭點證物，截止 2026-06-12')`（系統不自動推播；通知頻道選擇見 line-comms、多 OA 別寫死 `default`）

### 完成任務
1. `update_task(task_id=3, status='done')`
2. `log_interaction(actor='助理甲', action='task_completed', target_type='task', target_id=3, detail='完成 影印爭點證物')`

## 中斷恢復

如果 context 被壓縮：
1. `get_context_summary(scope='full')` — 查看「待處理任務」區塊，含優先級和截止日
2. `list_tasks(status='in_progress')` — 找出所有進行中的任務
3. 根據優先級和截止日排序，從最緊急的開始處理

## 十、注意事項

- task 不承接法定時限：再強調一次，法定期間一律 legal-admin `create_deadline`（確定性計算附 `statutory_basis`），task.due_date 只是內部行政提醒
- 不用子任務，保持扁平
- 每次狀態變更後 agent 接著 `log_interaction`（`update_task` 不會自動寫 interaction_log）
- 找不到任務時用 `search_tasks` 模糊搜尋
- LINE 通知保持簡短，一則不超過 3 行
- 批量建立任務時統一報告，非逐一回覆
