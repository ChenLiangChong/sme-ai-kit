# 任務管理專業指南

## 觸發情境

「建立任務」「指派給員工甲」「我的待辦」「進度如何」「做完了」「任務列表」
「這週要做什麼」「哪些逾期了」「幫我排優先」「會議紀錄轉任務」

---

## 一、任務建立流程

### 從對話中提取任務

1. 從對話中提取：標題、描述、指派對象、優先級、截止日期、category（分類）
2. 如果有指派對象，先 `lookup_employee` 確認人存在
3. 呼叫 `create_task`
4. 指派對象有綁 LINE → 用 `reply` 通知：「你有一項新任務：{標題}，截止 {日期}」

### category 分類

`create_task` 支援的 category 值：

| category | 用途 |
|----------|------|
| `general` | 一般任務（預設） |
| `rock` | 季度 Rock（EOS 框架，見 ops-dashboard） |
| `production` | 生產相關 |
| `delivery` | 配送相關 |
| `admin` | 行政庶務 |
| `meeting` | 會議行動項目 |

缺少資訊時主動問，不要猜：
- 沒說指派給誰 → 「這個任務要指派給誰？」
- 沒說截止日 → 「有截止日期嗎？」
- 沒說優先級 → 預設 normal，不用問

### 任務分解框架（大專案拆成小任務）

當使用者描述的是一個大目標（例：「開新店」「辦尾牙」「改裝修」），主動進行拆解：

**拆解原則 — SMART 任務標準：**
- **S** 明確（Specific）：做什麼很清楚，不模糊
- **M** 可衡量（Measurable）：完成時有明確產出物
- **A** 可指派（Assignable）：能交給一個人負責
- **R** 合理（Realistic）：1-5 個工作天能完成
- **T** 有時限（Time-bound）：有截止日期

**拆解步驟：**
1. 確認最終目標和期限
2. 列出所有必要步驟（大約 5-15 個）
3. 識別先後依賴關係（哪些要先做完才能做下一步）
4. 為每個步驟估時間
5. 逐一建立任務，在描述中註明依賴關係

**拆解範例 —「辦員工旅遊」：**

| # | 任務 | 負責人 | 天數 | 依賴 |
|---|------|--------|------|------|
| 1 | 調查員工時間偏好 | HR | 3天 | — |
| 2 | 蒐集 3 個方案報價 | HR | 5天 | #1 |
| 3 | 老闆核准方案和預算 | 老闆 | 2天 | #2 |
| 4 | 確認訂房和交通 | HR | 3天 | #3 |
| 5 | 通知全員行程細節 | HR | 1天 | #4 |
| 6 | 收集特殊需求（素食、過敏） | HR | 3天 | #5 |

當使用者只說「幫我建個任務：辦員工旅遊」→ 回覆：
「這是個大專案，我建議拆成以下步驟……你覺得這樣合理嗎？確認後我逐一建立。」

## 二、優先級判斷框架

### 艾森豪矩陣（適用 SME 版）

當使用者問「幫我排優先」或有超過 5 個待辦需要排序時：

|  | 急迫-高 | 急迫-低 |
|---|---------|---------|
| **重要-高** | ① 立刻做（今天處理） | ② 排時間做（本週完成） |
| **重要-低** | ③ 交給別人做（委派） | ④ 考慮不做（觀察清單） |

**判斷「重要性」的準則：**
- 高：影響營收、影響客戶滿意度、有法律/合規要求、老闆親自交辦
- 低：內部流程優化、nice-to-have、學習性質

**判斷「急迫性」的準則：**
- 高：今明天到期、客戶在等、有外部時限（報稅、合約）
- 低：截止日在一週以上、沒有外部壓力

**自動標記邏輯：**
- 截止日 ≤ 2 天且重要 → 自動升為 `urgent`
- 已逾期 → 一律標為 `urgent`
- 無截止日且 low → 每兩週提醒：「這個任務還需要嗎？」

### 每日任務建議排序

當使用者問「今天要做什麼」，按以下順序排列：
1. 已逾期的任務（紅色警示）
2. 今天到期的任務
3. urgent 且 in_progress
4. urgent 且 pending
5. normal 且今天到期
6. 其餘按截止日排序

## 三、狀態工作流

### 標準狀態流轉

DB 支援的狀態值：`pending` | `in_progress` | `done` | `cancelled`

```
pending → in_progress → done
  │
  └→ cancelled
```

### 卡住（blocked）處理

DB 沒有 `blocked` 狀態。當任務被阻擋時：
1. 狀態維持 `in_progress`（或視情況退回 `pending`）
2. 在描述前方加註：`[BLOCKED] 原因：{原因}`
3. 通知建立者 + 主管
4. 阻擋解除後移除 `[BLOCKED]` 標記
5. 查詢所有被擋任務：`search_tasks` 搜尋描述含 `[BLOCKED]`

### 狀態操作規則

| 動作 | 觸發語句 | 操作 | 後續 |
|------|---------|------|------|
| 開工 | 「開始做了」「我在處理#3」 | `update_task(id, status='in_progress')` | 通知建立者 |
| 完成 | 「做完了」「#3 完成了」 | `update_task(id, status='done')` | 通知建立者 |
| 卡住 | 「等材料」「被擋住了」 | 維持 `in_progress`，描述加註 `[BLOCKED] {原因}` | 通知建立者+主管 |
| 取消 | 「取消#5」「不做了」 | 先確認 → `update_task(id, status='cancelled')` | 通知相關人 |

## 四、到期提醒

由 CLAUDE.md 啟動流程觸發，查所有有 due_date 且未完成的任務：

| 時間點 | 動作 | 通知對象 |
|--------|------|---------|
| 到期前 2 天 | 提醒 | 負責人 |
| 到期前 1 天 | 強調提醒 | 負責人 |
| 到期當天 | ⚠️ 警告 | 負責人 |
| 逾期 1 天 | 逾期通知 | 負責人 + 建立者 |
| 逾期 3 天 | 升級通知 | 負責人 + 建立者 + 主管 |
| 逾期 7 天 | 強制檢討 | 主管，建議取消或重新排期 |

## 五、每週回顧

當使用者說「週回顧」「這週做了什麼」，執行：

**資料收集：**
- `list_tasks(status='done')` → 本週完成
- `list_tasks(status='pending')` → 待辦
- `list_tasks(status='in_progress')` → 進行中

**產出格式：**
```
第 XX 週回顧（MM/DD - MM/DD）

✅ 本週完成：X 項
🔄 進行中：X 項
⏳ 待辦：X 項
🔴 逾期：X 項

完成率：XX%
準時率：XX%
```

**健康指標：**

| 指標 | 健康 | 警告 | 危險 |
|------|------|------|------|
| 週完成率 | ≥ 80% | 60-80% | < 60% |
| 準時率 | ≥ 70% | 50-70% | < 50% |
| 逾期任務數 | 0-2 | 3-5 | > 5 |

## 六、委派原則（20-30 人團隊）

### 工作量平衡檢查

指派任務前，檢查該員工目前的工作量：

| 進行中+待辦數量 | 建議 |
|-----------------|------|
| 0-3 項 | 正常指派 |
| 4-6 項 | 提醒老闆：「{員工}目前有 X 項任務」 |
| 7+ 項 | 建議重新分配 |

### 委派品質檢查

建立任務時，確認描述包含：
- [ ] 期望的產出物
- [ ] 截止時間
- [ ] 完成標準
- [ ] 遇到問題找誰

## 七、會議 → 行動項目

當使用者說「整理會議紀錄」「剛開完會」：

1. 抽取所有「誰要做什麼、什麼時候完成」
2. 只有明確有負責人+動作的才建任務
3. 排除純資訊分享、待討論、FYI
4. 逐一建立任務，統一通知被指派的人

| 會議中的說法 | 轉換 |
|------------|------|
| 「員工甲你去問報價」 | 任務：詢問報價，指派員工甲 |
| 「下週前企劃交出來」 | 任務：完成企劃，截止下週五 |
| 「這個之後再說」 | 不建任務 |
| 「大家回去想想」 | 不建任務（無明確負責人） |

## 八、常見陷阱

| 陷阱 | 處理 |
|------|------|
| 口頭交辦不記錄 | 提醒：「要不要建個任務追蹤？」 |
| 任務描述太模糊 | 主動問：「完成標準是什麼？」 |
| 全部都很急 | 套用優先級矩陣強制排序 |
| 一人扛太多 | 工作量平衡檢查 |
| 只建不關 | 週回顧時清理 |
| pending 超 14 天 | 提醒：「這個任務是不是可以取消？」 |
| in_progress 超 7 天 | 追蹤：「進度如何？需要幫忙嗎？」 |

## Do's and Don'ts

### Do
- 缺少資訊主動問（指派對象、截止日），不要猜
- 每次狀態變更自動 `log_interaction`
- `in_progress` 超 7 天主動追蹤：「進度如何？需要幫忙嗎？」
- `pending` 超 14 天提醒：「這個任務是不是可以取消？」
- 指派前檢查員工工作量（4-6 項提醒老闆，7+ 項建議重新分配）

### Don't
- 不要猜 deadline（沒說就問「有截止日期嗎？」）
- 不要用子任務（Phase 1 保持扁平）
- 不要逐一回覆批量建立的結果（統一報告）
- 不要把模糊的「之後再說」「大家回去想想」建成任務
- 不要在 LINE 通知中超過 3 行

## 快速參考

### 建立任務
1. `lookup_employee(name_or_line_id='員工甲')` — 確認人存在
2. `create_task(title='詢問報價', assignee='員工甲', priority='normal', due_date='2026-04-10', category='admin', created_by='老闆')`
3. `reply(channel_id='default', chat_id=員工甲的LINE_user_id, text='你有一項新任務：詢問報價，截止 4/10')`

### 完成任務
1. `update_task(task_id=3, status='done')`
2. `log_interaction(actor='員工甲', action='task_completed', target_type='task', target_id=3, detail='完成 詢問報價')`

## 中斷恢復

如果 context 被壓縮：
1. `get_context_summary(scope='full')` — 查看「待處理任務」區塊，含優先級和截止日
2. `list_tasks(status='in_progress')` — 找出所有進行中的任務
3. 根據優先級和截止日排序，從最緊急的開始處理

## 十、注意事項

- Phase 1 不用子任務，保持扁平
- 每次狀態變更自動寫 interaction_log
- 找不到任務時用 search_tasks 模糊搜尋
- LINE 通知保持簡短，一則不超過 3 行
- 批量建立任務時統一報告，非逐一回覆
