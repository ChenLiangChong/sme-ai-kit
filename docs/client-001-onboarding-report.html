一、系統設計哲學

通用模組 + 深度訪談客製化

SME-AI-Kit 不是為特定產業「寫死」的系統。它提供 12 個通用營運模組（任務、庫存、帳務、客戶、訂單、LINE 通訊等），透過深度訪談把每家公司的規則、術語、流程灌進資料庫。同樣的系統，餐廳用「桌號」，日誠用「經銷商」，差別不在程式碼，而在老闆教給 AI 的知識。

knowledge-capture 作為導入引擎

系統導入不是工程師寫程式，而是跟老闆聊天。knowledge-capture 模組定義了 9 步標準流程：從公司基本資料、員工名冊、客戶清單、商品庫存、品牌語氣，到深度訪談和校準測試。每條規則都附上老闆原話（source_quote），確保 AI 記住的是老闆說的，不是 AI 自己編的。

HITL（Human-In-The-Loop）審核機制

AI 不是自己做主。超過金額門檻的支出、對外行銷訊息、批次修改客戶資料，都必須先建立審核單（create_approval），等老闆在 LINE 上回覆「核准」才執行。老闆隨時有最終決定權。

反捏造原則

這是系統最嚴格的紀律：

  • 老闆親口說的 → 標記 source_type='explicit'，附上原話
  • AI 觀察到的慣例 → 標記 source_type='observed'，告知老闆
  • AI 推斷的 → 標記 source_type='inferred'，明確標注「待確認」
  • 絕對不把推斷偽裝成老闆的指示

為什麼不針對特定產業寫死？

台灣中小企業千變萬化 — 賣蠟燭的、做裝潢的、開咖啡廳的，每家都不一樣。但核心需求驚人地相似：接單、出貨、收錢、管人、管庫存、跟客戶溝通。

SME-AI-Kit 把這些共通需求模組化，再透過 knowledge-capture 把每家公司的「獨特之處」灌進去。這樣一套系統可以服務很多種公司，而不是每家都從頭寫。

四、導入流程對照

以下是 knowledge-capture 的 9 步標準導入流程，具體化到日誠的情境。不需要一次做完，可以分 2-3 天，老闆累了就停。

1

公司基本資料

用 store_fact(category='company') 存入：

問什麼日誠的答案（預填參考）
公司名稱日誠有限公司
品牌名稱In Love With My Space（WIA）
個人品牌拓荒者 York
產業別居家設計商品 / B2B 經銷
公司規模待訪談確認
主要業務居家商品設計 → 工廠生產 → 全台經銷 + 內容行銷
核心痛點訂單追蹤混亂、應收帳款催不回來、內容排程跟不上
2

員工名冊 + LINE 綁定

用 register_employee 逐一建入。老闆（York）必須第一個綁定。

register_employee(
  name='York',
  role='boss',
  department='管理部',
  permissions='admin',
  phone='...'
)
// York 傳一則 LINE 訊息 → lookup_employee 配對 → 綁定 line_user_id

// 其他員工依序建入（設計師、倉管、業務等）
register_employee(
  name='[員工名]',
  role='staff',
  department='[部門]',
  permissions='basic',
  phone='...'
)

需要 York 提供：完整員工名單（姓名、職位、手機）。每位員工需傳一則訊息到 LINE OA 完成綁定。

3

經銷商 / 供應商清單

用 add_customer 逐一建入。先建最常往來的 5-10 家，後續邊用邊補。

// 經銷商範例
add_customer(
  name='好好生活家居',
  type='distributor',
  contact_person='李經理',
  phone='02-2345-6789',
  notes='月結 30 天，每月固定進貨 WIA-001/002'
)

// 工廠（供應商）範例
add_customer(
  name='XX 蠟燭工廠',
  type='supplier',
  contact_person='陳老闆',
  phone='04-1234-5678',
  notes='打版費 NT$3,000/版，交期約 14 天'
)

需要 York 提供：經銷商名稱、聯絡人、電話、帳期條件。供應商名稱、交期、付款條件。

4

商品 SKU 清單

用 update_stock 逐一建入。如果商品多，老闆傳一張 Excel 截圖，AI 辨識後逐筆建入。

// WIA 商品範例
update_stock(sku='WIA-001', quantity=50, reason='初始庫存建檔')
update_stock(sku='WIA-002', quantity=30, reason='初始庫存建檔')
// ...更多品項

// 設定安全庫存門檻（存入知識庫）
store_fact(
  category='inventory',
  title='WIA-001 安全庫存',
  content='安全庫存 20 個，低於此數量要通知老闆補貨',
  source_type='explicit',
  source_quote='WIA-001 庫存低於 20 就要跟工廠下單'
)

需要 York 提供：所有在售商品的 SKU / 品名 / 目前庫存數量 / 安全庫存建議。

5

品牌語氣設定

透過 brand-voice 模組設定。推薦方式：請 York 提供 3-5 篇過去寫得好的文案。

// AI 分析後存入
store_fact(
  category='brand',
  title='WIA 品牌語氣',
  content='溫暖但不矯情。用「你」不用「您」（對消費者）。
    常用詞：生活感、空間氛圍、回家的感覺。
    避免：過度銷售語氣、「限時特惠」類用語。
    情感基調：像朋友分享好東西。',
  source_type='explicit',
  source_quote='（從老闆提供的文案範例中萃取）'
)

// 拓荒者 York 的個人語氣（不同於品牌）
store_fact(
  category='brand',
  title='拓荒者 York 語氣',
  content='真誠分享、不怕展現摸索過程。
    口頭禪/慣用語：（待訪談）。
    YouTube 風格：教學 + 生活分享。',
  source_type='inferred'  // 待老闆確認
)

需要 York 提供：3-5 篇滿意的文案/貼文/影片描述。品牌禁用詞清單。

6

深度訪談 — 日誠特有的規則

用 knowledge-capture 的主動訪談框架，逐一了解日誠的經營規則：

領域要問 York 的問題存入分類
帳期「經銷商的付款條件？月結幾天？不同經銷商不同嗎？」finance
催款「逾期多久開始催？怎麼催？催幾次不付怎麼辦？」finance
審核門檻「多少金額以上需要你親自核准？」finance
定價「經銷價怎麼算？不同經銷商不同價嗎？量大有折扣嗎？」pricing
QC 標準「出貨前怎麼檢查品質？什麼情況不能出？」sop
退貨「經銷商可以退貨嗎？條件？期限？瑕疵品怎麼處理？」return_policy
供應商「工廠延遲交貨怎麼處理？打版不滿意怎麼溝通？」supplier
報表「你每天/每週想看什麼數字？營收？庫存？訂單數？」general
內容行銷「YouTube 更新頻率？每週幾部？哪些平台一定要發？」sop

每條回答都走被動捕捉流程：確認 → 附上原話（source_quote）→ 存入。

7

業務流程映射

請 York 用白話描述三條業務線的完整流程，然後一一對照系統模組：

「York，你描述的 B2B 流程是：
  客戶 LINE 下單 → 確認庫存 → 出貨 → 月結收款

  這對應到系統的：
  1. LINE 訊息進來 → line-comms（辨識身份）
  2. 確認庫存 → inventory-ops（check_stock）
  3. 建立訂單 → order-ops（create_order）
  4. 品檢出貨 → order-ops（qc_order → fulfill_order）
  5. 月結收款 → accounting-ops（check_overdue → record_payment）

  這樣對嗎？有沒有漏掉的步驟？」

確認後的流程用 store_fact(category='sop') 存入。

8

校準測試 — 拿一張真實訂單跑一輪

找一張真實的經銷商訂單，從頭到尾跑一遍：

  • 模擬經銷商 LINE 下單 → AI 辨識身份 → 建單 create_order
  • 確認庫存 → check_stock
  • 品檢 → qc_order(result='passed')
  • 出貨 → fulfill_order（自動扣庫存 + 建應收帳款）
  • LINE 通知客戶 → reply
  • 模擬收款 → record_payment
  • 把結果給 York 看：「這樣對嗎？哪裡需要調整？」

調整的部分回到 Step 6 補充規則。

七、系統限制與誠實揭露

以下是目前系統的限制。不誇大能力，不隱瞞不足。

1.
LINE MCP 沒有 quota 查詢功能
LINE 官方帳號有月訊息量限制（免費方案 200 則/月），但系統無法自動查詢剩餘額度。老闆需要手動到 LINE OA 後台確認使用量，避免超量被收費。
2.
✅ 已新增 update_employee 工具
可以直接更新員工資料：update_employee(id, phone='新號碼')。支援更新姓名、角色、部門、LINE 綁定、權限、電話、在職狀態、備註。離職可用 update_employee(id, active=0)。
3.
✅ 已支援 Excel / PDF / Word 匯出
系統已整合 office skills（xlsx / pdf / docx / pptx），可直接產出 .xlsx 報表、.pdf 報價單、.docx 合約等檔案。老闆說「匯出 Excel」就會產出檔案。
4.
社群 MCP 目前只有蒐集功能
可以蒐集 FB / IG / Threads 的貼文、留言、互動數據。但發文功能暫時停用，發佈貼文仍需老闆手動操作各平台。
5.
YouTube / TikTok MCP 尚未支援
這兩個平台是拓荒者 York 內容漏斗的關鍵環節，但目前系統無法直接讀取或操作。只能用 task-ops 追蹤任務進度、用 copywriting 模組幫忙寫文案。
6.
AI 不能替代老闆的設計專業判斷
打版品質好不好、顏色對不對、美學感覺如何 — 這些全靠 York 的專業眼光。AI 只能追蹤進度和記錄意見，不能替老闆做設計決策。
7.
AI 不能替代老闆的關係經營
經銷商的應酬、工廠的價格談判、新客戶的信任建立 — 這些都是人對人的工作。AI 管資料和流程，老闆管關係和決策。
8.
✅ 已新增 LINE multicast 群發功能
multicast(user_ids='["U1","U2",...]', text='訊息') 可一次發送給最多 500 人。搭配 crm-ops 篩選客群後一鍵群發。
