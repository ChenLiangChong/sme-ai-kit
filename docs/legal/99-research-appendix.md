## 二、上訴理由書「補提期間」（空白上訴的續命期，務必獨立列管）

上訴與「補提上訴理由書」是**兩條獨立的時鐘**，律所最常漏的是第二條。

| 程序別 | 動作 | 期間 | 起算事件 | 性質 | 法條依據 |
|---|---|---|---|---|---|
| 民事三審 | 上訴狀未表明理由→補提理由書於原二審法院 | **20 日** | 提起上訴後（逾期毋庸命補正，原二審法院逕以裁定駁回） | 法定期間（律師強制代理；未委任律師前法院應先命補正、不得逕駁） | 民訴 §471 I |
| 刑事二審 | 上訴狀未敘述理由→補提理由書於原審法院 | **20 日** | 上訴期間屆滿後（逾期未補→原審法院定期間先命補正） | 法定期間 | 刑訴 §361 III |
| 刑事三審 | 上訴狀未敘述理由→補提理由書於原審法院 | **20 日** | 提起上訴後（逾期毋庸命補提） | 法定期間 | 刑訴 §382 I |
| 行政三審 | 上訴未表明理由→補提理由書（準用民訴 §471 模式） | **20 日** | 提起上訴後 | 法定期間（上訴審強制律師代理，現行依行訴 §49-1） | 行訴 §245 準用 |
| 民事二審 | （無）上訴理由補提 | **無法定硬性補提期間** | — | 任意提出主義：未提理由不得逕駁，準用 §447、判決時依全辯論意旨斟酌 | 民訴 §444-1、§447 |

## 三、性質分類與實務判讀（給時限引擎的規則）

**1. 不變期間（peremptory，過了即失權、法院不得伸長縮短）**
- 上訴 20 日 / 抗告 10 日 / 再審之訴 30 日 / 支付命令異議 20 日 / 行政撤銷訴訟起訴 2 個月，及其各程序的對應期間。
- 唯一救濟＝回復原狀（民事 / 行政 10 日、刑事 10 日），且須「非因過失」+ 補行原訴訟行為，遲誤逾 1 年（民事）即不得聲請。**回復原狀是事後補救、不是可依賴的緩衝，內部期限仍應以原不變期間為準。**

**2. 法定期間（一般，可命補正、未必立即失權）**
- 上訴理由書補提 20 日：刑訴二審逾期會「先命補正」（有第二次機會）；但民訴三審 §471 / 刑訴三審 §382 逾期「毋庸命補提、逕駁」，實務上應比照不變期間嚴管。
- 訴願 30 日：性質上是法定期間，但漏掉＝喪失行政救濟，務必當不變期間管。

**3. 訓示 / 期日性期間（對律所時限管理風險低，僅供完整）**
- 法院端的「裁判應於 X 日內宣示 / 送達」、勞動調解「3 個月 3 次期日內終結」屬訓示性、對律所遞狀無失權效果，不入失權倒數，但可做進度提醒。

## 四、起算與計算規則（時限引擎必須內建，否則算錯＝過失）

- **起算翌日**：期間自送達 / 確定之**次日**起算（民法 §120 II，刑訴 §65 準用民法）。
- **末日遇假日順延**：末日為星期日 / 國定假日 / 其他休息日→以次一工作日為末日（民法 §122）。**注意：只有「末日」順延，期間「之內」的假日照常計入、不扣除。**
- **在途期間加計**：當事人 / 訴訟代理人不在法院所在地住居者，依司法院「法院訴訟當事人在途期間標準」**加計**在途日數於法定期間外（民訴 §162、刑訴 §66）。是否加計以「訴訟代理人是否在法院所在地 + 是否有特別代理權」判定。**這是時限引擎最易算錯的一塊，建議內部期限再扣安全緩衝。**
- 司法院提供官方「上訴 / 抗告期間試算工具」（gdgt.judicial.gov.tw），已含在途期間與末日假日參數，可作為自動算出結果的交叉驗證來源。

## 五、給 legal-admin vertical 的工程建議（system-layer 時限管理）

1. **「19 天」= 20 日不變期間 − 1 安全緩衝** 的假設，與本研究完全吻合：上訴（民/刑/行/家）全是 20 日不變期間。建議內部期限不只 −1，而是**從「送達日」起算後同時建三個倒數**：(a) 安全內部期限（如 −3 工作日）、(b) 法定不變期間末日（含在途期間加計、末日假日順延）、(c) 補提理由書 20 日獨立時鐘（若採空白上訴）。
2. **system-layer 最適合「不變期間」**：因為它時間驅動、與人是否開 Claude 無關，正可比照既有 escalation 基礎設施——`flush_escalations.py`（OS cron 保證層）每日掃「距末日 N 日」的 deadline row 主動推播 + `claude -p` single-shot 品質層生成提醒文字 + in-session push 即時注入。
3. **失權型 vs 可補正型分流**：不變期間 + 三審補提（§471/§382 逾期逕駁）走「紅色硬倒數、雙人覆核」；刑訴二審補提（§361 先命補正）、訴願 30 日走「橘色」；訓示期間走「灰色僅提醒」。
4. **必做的法規版本鎖定**：刑訴上訴期間（10→20 日，2021）、刑訴抗告 + 回復原狀（5→10 日，2023）兩次修正若引用舊資料會直接算錯。時限引擎的法條基準應標註版本日期並定期以全國法規資料庫覆核。

## 0. 先理解：律所行政的「兩種時鐘」

律所行政可拆成兩種根本不同性質的工作，這決定了 legal-admin vertical 的架構分層：

| 類型 | 性質 | 例子 | 適合的執行層 |
|------|------|------|------------|
| **事件驅動（event-driven）** | 有人觸發才發生 | 接案、開委任、記時數、出帳單、利衝檢查 | 互動 session（老闆 / 律師助理打開 Claude 才做） |
| **時間驅動（time-driven）** | 時間到就發生，沒人也在倒數 | 上訴期限、答辯狀期限、開庭前提醒、結案保存年限到期 | **system 層**（OS cron + claude -p single-shot），不依賴有人開 session |

**老闆架構主張的研究支持**：時限管理本質上是 time-driven —「判決送達後 20 日內不上訴 = 喪失上訴權」這個倒數，**律師沒開 Claude 也在跑**。這正是既有 escalation 基礎設施（`flush_escalations.py` cron 保證層 + claude -p notifier 品質層 + in-session push 即時層）的完美第二個應用：把「pending_escalations 笨投遞」的模式，複製成「pending_deadlines 笨倒數 + 多閾值提醒」。這是本 vertical 的技術核心，不是附加功能。

## 1. 案件 / 案號管理（matter management）

**是什麼**
律所的核心物件是「案件（matter / 案子）」，不是「訂單」。一個 matter 從接案到結案有完整生命週期，且**會跨多個審級**。

**生命週期（階段欄位）**
台灣訴訟案件典型階段流轉：
- **偵查階段**（刑事）：地檢署，案號用「偵」字
- **一審**：地方法院，民事訴訟用「訴」字、刑事用「訴 / 易」字
- **二審**：高等法院（上訴 / 抗告）
- **三審**：最高法院（法律審）
- **執行階段**：強制執行（「執」字）、或刑事執行

非訟 / 顧問類 matter（合約審閱、法律意見、商務談判）則沒有審級，是另一種 lifecycle。

**核心欄位（這是 matter 資料表的 schema 雛形）**
- 當事人（委任人，可多人）
- 對造（對方當事人，**利衝檢查的關鍵欄位**）
- 案由（請求權基礎 / 罪名，如「給付買賣價金」「詐欺」）
- 法院（管轄法院）
- **案號**：結構 = `年度 + 字別 + 流水號`，例「112 年度訴字第 1234 號」（112 年度＝民國年；訴字＝民事一般訴訟；1234＝序號）— 依據：臺灣高等法院以下各級法院民刑事案件案號字別對照表
- **股別（承股）**：法院內部分案單位，每位法官 / 檢察官負責一股，名稱五花八門（忠孝仁愛、天地玄黃、慈誠勝等千字文 / 吉祥字），向法院報案件時簡稱「承股」。**這是台灣特有欄位，一般案件管理軟體沒有**
- 承辦律師
- 階段（偵查 / 一審 / 二審 / 三審 / 執行）
- **關鍵**：案號與股別**會隨審級改變**（上訴後換法院、換案號、換股別），所以 matter 物件下要掛「審級子紀錄」，不是單一案號欄位

**痛點**
- 同一個 matter 在不同審級有不同案號 / 股別 / 承辦法官，紙本卷宗封面常貼便利貼層層疊加
- 委任**每審級都要重新提委任狀**（民訴規定「委任應於每審級為之」），漏提 = 該審級沒有合法代理人 → 這本身就是一個時限 / 流程提醒點
- 一人律師同時跑 20-50 個 matter，每個進度不同，全靠腦袋 + Excel

**現在怎麼做**
- 最常見：**Excel 案件總表**（一列一案，欄位手刻）+ 紙本卷宗
- 有預算的：專用軟體（康瓏、E律師、世盈穩盈、LEXICO、EZip 法律期限與工時系統、益盛）— 但這些多為**買斷 / 年費的 Windows 桌面系統**，一人所常嫌貴 / 難用 / 要自己 key

## 2. 當事人管理

**是什麼**
管理委任人（client）與相關方資料。當事人分**自然人**與**法人**（公司 / 機關 / 社團），欄位需求不同。

**核心欄位**
- 自然人：姓名、身分證字號、地址、聯絡電話、與案件關係
- 法人：公司名稱、統編、負責人、代表人、地址、聯絡窗口
- **委任關係**：哪個當事人委任哪些 matter、委任範圍（一般委任 vs 特別委任）。民訴規定捨棄、認諾、撤回、和解、提起上訴 / 再審、選任複代理人等「處分性行為」**非經特別委任不得為之** → 委任狀勾選欄位要記錄

**痛點**
- 同一當事人多個案件、同一案件多個當事人（共同被告 / 共同原告），多對多關係
- 當事人地址錯 → 影響「在途期間」計算（住居所不在法院所在地要加在途期間，連動上訴期限算法）
- 法人窗口換人、聯絡資訊散落

**現在怎麼做**
- Excel 通訊錄、紙本委任狀檔案夾、名片盒、手機通訊錄。極少有結構化的「當事人 ↔ matter ↔ 委任範圍」關聯表。

**架構意涵**：這幾乎可直接沿用既有 `customers` 模組（自然人 / 法人都有先例），但要把「customer」概念對映成「client」，並新增「對造」這個角色（利衝檢查要用）。

## 3. 利益衝突檢查（conflict check）— 律師倫理硬要求

**是什麼**
**接案前**，查詢欲委任的當事人 / 對造，是否與事務所**既有或過去**的案件當事人 / 對造產生衝突。例：不能同時代理夫妻離婚雙方；不能代理 A 告 B，又接 B 的另案委任。

**為何是倫理硬要求（不是 nice-to-have）**
- 依據：**律師法第 34 條** + **律師倫理規範第 30 條**。律師對於有利益衝突的事件**不得受任**；若構成第 30 條第 1 項各款情形，須告知受影響的現 / 前委任人並取得**書面同意**才能續任
- **同一事務所連坐**：律師倫理規範規定，一位律師受利衝限制者，**同事務所其他律師亦受相同限制**（利衝會「傳染」整個所）
- 違反 = 付律師懲戒（移送律師懲戒委員會），輕則申誡、重則停止執業 / 除名，且可能構成民事賠償
- 「實質關連性判斷」：對前委任人的衝突，看新案件與舊案件是否有實質關連

**痛點**
- 一人所開業久了，當事人 / 對造名單上百上千，**接新案前要靠記憶 + 翻舊卷**判斷有沒有衝突 → 極易漏
- 對造的「關係企業 / 關係人」也要查（B 公司的負責人是不是我以前的當事人）
- 這是**最該自動化卻最少自動化**的一塊（純文字比對 + 關係圖譜，恰好是 DB query 的強項）

**現在怎麼做**
- 小所多半**靠律師記憶 + 翻紙本卷宗目錄**，幾乎沒有系統化比對
- 有系統的所會「跑 conflict check」：把新當事人 / 對造姓名丟進案件資料庫比對
- **架構意涵**：這是 legal-admin 一個高價值、低風險（只讀、只建議、最終由律師判斷）的 tool —「接案前 `conflict_check(當事人, 對造)` → 掃 matter 表的所有 client / 對造欄位 → 列出潛在衝突 + 關連案號讓律師判斷」。可做成接案流程的強制 gate（類似既有 HITL）。

## 4. 計時計費（billable hours）

**是什麼**
按時計費（鐘點制）的案件，律師要逐筆記錄「在這個 matter 上花了多少時間做什麼」，月底彙總成帳單向客戶請款。

**運作**
- 一筆 time entry = `matter + 日期 + 時長（常以 6 分鐘 = 0.1 小時為最小單位，沿用國際慣例）+ 工作內容描述 + 費率`
- 帳單 = 該 matter 某期間所有 time entry 加總 × 費率 + 代墊費用（規費、影印、車馬）
- 費率：依律師年資 / 案件複雜度，台灣鐘點行情約 **NT$3,000–6,000 / 小時**，複雜案件可達 1.2 萬以上；夜間 / 假日有 1.2–2 倍加成 — 依據：2026 律師費行情整理

**痛點**
- **記時是律師最討厭、最容易漏的行政**：做的當下沒記，事後回想 = 漏算 = 少收錢（漏記的時數是真金白銀蒸發）
- 一人所常「做完才回想這個月花多少時間」→ 嚴重低估
- 工作描述要寫得讓客戶看得懂又願意付（「研究 X 爭點 2 小時」vs「處理案件」）

**現在怎麼做**
- Excel 工時表、紙本記事、或乾脆不記（改採按件計費規避記時麻煩）
- 專用軟體有計時模組（康瓏標榜 Hourly Rate 計費、EZip 主打工時管理），但一人所嫌麻煩
- **架構意涵**：可做成 LINE 快速記時（「在 [案號] 花了 1.5 小時研讀對造答辯」→ 自動建 time entry），降低記錄摩擦；月底 `generate_bill(matter, 期間)` 彙總。對映既有 `transactions` / 任務時數概念。

## 5. 律師費收費方式（台灣行情概念）

**五種主要模式** — 依據：2026 律師費行情整理、政理 / 喆律 / 好律師等行情頁

| 模式 | 說明 | 台灣行情概念 |
|------|------|------------|
| **按件（論件計酬）** | 單一文書 / 聲請 | 函件 NT$3,000 起；簡單契約 3,000–10,000；正式契約 1–6 萬；支付命令 / 本票裁定 8,000–3 萬；遺囑撰稿 1.5 萬起 |
| **按審級（包審）** | 走完一個審級 | 約 **NT$8 萬–20 萬 / 審**（偵查、一審、二審、三審各算一審級） |
| **按時（鐘點）** | 諮詢 / 顧問 / 難估工作量 | NT$3,000–6,000 / 小時，複雜案 1.2 萬+ |
| **後酬（成功報酬）** | 先付小額底費，成功後按比例抽成 | 適合高額民事求償（標的 500 萬+）；**倫理硬限制：律師倫理規範第 35 條 + 律師法第 34 條 —家事、刑事、少年事件不得約定後酬** |
| **顧問月費 / 年費** | 常年法律顧問 | 一年一聘，含一定次數諮詢；專案顧問月費 NT$30,000 起 |

**痛點**
- 同一所內不同 matter 用不同收費模式 → 帳務系統要同時支援「按審級的固定金額分期」「按時的彙總」「後酬的條件式」
- 後酬要追蹤「案件結果」才知道能不能收 / 收多少
- **諮詢費先收 vs 委任費分期**：實務上絕大多數律師「事先收費」，委任費常分階段（簽約付一半、開庭付一半）

**架構意涵**：收費模式是 matter 的屬性，連動帳單生成邏輯。後酬模式要把「成功條件」存成可追蹤狀態。倫理禁止後酬的案件類型要做成**接案時的硬檢核**（family/criminal/juvenile + 後酬 = 擋下並提示違反第 35 條）。

## 6. 信託帳 / 代收代付（client trust account 在台灣的對應）

**重要研究發現：台灣沒有美國式的強制 client trust account（IOLTA）制度。**

- **美國**：律師代收和解金 / 賠償金 / 保留金，**強制**存入與事務所自有資金分離的 client trust account，受嚴格 fiduciary duty 規範，孳息歸法律扶助基金（IOLTA）。混用客戶款 = 重大違紀。
- **台灣**：**沒有對應的強制專戶法制**。但律師倫理規範有功能相近的義務：
  - 律師對受任事件**代領 / 代收之款項，應即時交付委任人**（不得遲延、不得挪用）
  - 律師對保管與事件有關之物品，事件完畢後應返還，不得無故遲延 / 拒絕

**實務上「代收代付」場景**
- 代收：和解金、判決賠償金、執行所得、對造給付的款項
- 代付：法院裁判費 / 執行費 / 規費、鑑定費、公證 / 認證費、登記規費等代墊款

**痛點**
- 台灣多數小所**沒有獨立信託專戶**，代收款常進事務所一般帳戶或律師個人帳戶 → 帳目混淆風險（雖無強制專戶，混用仍可能生倫理 / 民事爭議）
- 代墊費用要逐筆記錄、結案時與當事人結清（哪些已收回、哪些待請款）
- 「代收 X 萬還沒交付當事人」這種**待辦 + 時限**性質的款項追蹤，最容易出包

**現在怎麼做**
- Excel 代墊 / 代收明細表，或夾在案件帳裡。極少數會自設「客戶款專戶」。
- **架構意涵**：DB 要把「事務所自有收支」與「代客戶保管 / 代墊代收」**分帳記錄**（兩個 ledger），即使台灣不強制專戶，分帳本身就是賣點（合規 + 不出包）。「代收未交付」可做成主動提醒（time/event-driven 混合）。

## 7. 卷宗與文件管理

**是什麼**
管理一個 matter 的所有文件：書狀（起訴狀、答辯狀、準備書狀、上訴狀、聲請狀…）、證物、判決書 / 裁定、開庭通知、委任狀、往來函件、契約。

**典型文件分類**
- 自己提出的書狀（含版本控制 — 草稿 / 定稿 / 已遞）
- 收到的文件（法院送達、對造書狀、判決裁定）
- 證據 / 證物清單
- 行政文件（委任狀、收據、報價）

**痛點**
- 一個 matter 動輒數百頁、跨審級，紙本卷宗厚重；找「上次那份對造的答辯狀」靠翻
- 書狀要**版本控制**（遞出去的是哪一版）+ **與時限綁定**（這份答辯狀的遞交期限是哪天）
- 電子化後檔名混亂（「答辯狀.docx」「答辯狀最終版.docx」「答辯狀真的最終.docx」）
- 大所已開發電子卷宗系統做「紙本瘦身」（如 TIPLO 案例），小所仍多紙本

**現在怎麼做**
- **主流仍是紙本卷宗 + 電腦資料夾（按案件開資料夾）**
- 進階：掃描電子化、雲端硬碟（Google Drive / NAS）
- 專用軟體有書狀管理 / 收發文模組（康瓏「書狀管理系統」、E律師「書狀工作管理」、LEXICO「收發文系統」）
- **架構意涵**：對映既有 `attachments` 模組，但要加「文件類型 + 審級 + 版本 + 關聯時限」維度。書狀與「期日 / 期限」綁定是關鍵差異化。

## 8. 開庭 / 期日行事曆（與時限管理的關係 — 本 vertical 核心）

**是什麼**
管理法院通知的開庭期日（言詞辯論期日、準備程序期日、調查證據期日、宣判期日、偵查庭 / 訊問期日）+ 與之連動的**書狀提出期限**。

**關鍵：期日與時限是一體兩面（這是律所行政最致命的一塊）**

法院的**期日**（要到場）會反推出一連串**期限**（要在某天前遞書狀），各有不同法定 / 指定天數：

| 期限類型 | 天數 | 法條依據 |
|---------|------|---------|
| **民事上訴期間（二審）** | **20 日不變期間**，自判決送達翌日起算 | 民事訴訟法第 440 條 |
| **刑事上訴期間** | **20 日**，自送達判決起算（原為 10 日，**108.12.17 三讀、配合兩公約延長為 20 日**） | 刑事訴訟法第 349 條 |
| **民事抗告期間** | **10 日不變期間**，自裁定送達後起算 | 民事訴訟法第 487 條 |
| **被告答辯狀** | 收受訴狀後 **10 日內**提出；已定言詞辯論期日者，至遲**期日 5 日前** | 民事訴訟法第 267 條 |
| **準備書狀** | 收受答辯狀後 **5 日內**；已定期日者至遲**期日 3 日前** | 民事訴訟法第 267 條 |
| **刑事再議** | **10 日**（原 7 日，同 108 修法延長） | 刑事訴訟法第 256 條 |
| **補提上訴理由書（刑事）** | **20 日**（原 10 日，同 108 修法延長） | 刑事訴訟法第 382 條 |

**期間計算的魔鬼細節（演算法必須處理，不能憑印象）**
- **起算**：自送達**翌日**起算（首日不算入）
- **不變期間中間遇假日不扣除**（20 日含假日照算）
- **末日遇假日順延**：若期間末日為週日 / 國定假日 / 其他休息日，以次一上班日代替（依民法第 122 條）
- **在途期間**：當事人住居所不在法院所在地者，要**加計在途期間**（依各地在途期間標準表，可能加數日）— 這會改變實際 deadline，算法必須查當事人地址 vs 法院所在地
- **遲誤救濟**：非可歸責於己遲誤不變期間，原因消滅後 **10 日內**得聲請回復原狀，但逾 1 年不得聲請（民訴第 164 條）

**「19 天」的研究印證**：老闆查到的「19 天」極可能正是「法定 20 日 − 1 天安全緩衝」的內部期限 — 律所實務普遍會把**內部 deadline 設在法定期限前 1-3 天**，留出收件、確認、用印、遞狀的緩衝。這正驗證了「**多閾值倒數提醒**」的設計需求（法定日、內部緩衝日、最後通牒日各一個提醒）。

**痛點（這是律所的「漏一個＝執業過失」核心痛點）**
- 漏記一個上訴 / 答辯期限 → 喪失上訴權 / 失權效果 → **律師執業過失，要對當事人賠償 + 可能懲戒**
- 期日通知用紙本送達，律師助理要手動轉錄到行事曆 + 反推所有連動期限
- 一人所同時跑數十案，每案各自有期日 / 期限，**人腦 + Excel 極易漏**
- 在途期間、末日順延等算法細節容易算錯

**現在怎麼做**
- Google Calendar / 紙本行事曆 / 桌曆 + 律師助理人工反推期限
- 專用軟體有期限管理（**EZip 主打「法律期限與工時管理」、康瓏有「強制提醒機制」**）— 但仍是「要開軟體才看得到」
- **架構意涵（本 vertical 的核心價值主張）**：
  - 期限管理是 **time-driven**，天生屬於 **system 層**，不依賴有人開 Claude
  - 完全複製既有 escalation 三層架構：建一張 `pending_deadlines`（matter_id、期限類型、法定天數、起算日、法條依據、內部緩衝日、各閾值是否已提醒）
  - **OS cron（保證層）** 每日掃 `pending_deadlines`，到達各閾值（如 D-7 / D-3 / D-1 / 當日）就 enqueue 提醒，確定性投遞到 LINE
  - **claude -p notifier（品質層）** 產生有脈絡的提醒文字（「[112 訴字 1234 號] 答辯狀法定期限 6/15、內部緩衝 6/13，剩 3 天，對造主張…建議今天定稿」）
  - **in-session push（即時層）** 律師正開著 Claude 時即時跳出
  - 期間計算 tool 內建：翌日起算 + 假日順延 + 在途期間查表 + 內部緩衝，**每次回傳附法條依據**（呼應反捏造原則 — 法律天數不可憑印象）

## 9. 結案歸檔、保存年限

**是什麼**
案件終結（判決確定 / 和解 / 撤回 / 不起訴）後，將卷宗整理歸檔、設定保存年限、屆期銷毀。

**保存年限（研究發現：私人律所無單一法定統一年限）**
- **法院 / 檢察署檔案有明確基準**（法院類 / 檢察類檔案保存年限基準表）：定期檔案分 30/25/20/15/10/5/3/1 年；**有裁判的案卷多為永久保存**（這是法院端，非律所端）
- **私人律師事務所**：台灣**沒有像法院那樣的單一強制統一保存年限法規**。實務保存年限取決於：
  - **消滅時效 / 追訴期**：案件相關請求權時效（民事一般 15 年、租金等 5 年、侵權 2 年）內應留存以備爭議
  - **律師可能被告過失的舉證需要**：保留卷宗以證明自己無過失（建議至少留到相關時效屆滿後一段時間）
  - 委任契約如有約定保存 / 返還條款則從約定
  - 倫理規範：事件完畢應返還當事人之物品 / 文件，不得遲延拒絕
- 實務做法：多數所**自訂保存政策**（常見 5–10 年，重要案件更久 / 永久），到期前通知當事人領回或銷毀

**痛點**
- 紙本卷宗堆積佔空間 → 大所做「紙本瘦身電子化」，小所則塞倉庫 / 律師家裡
- 沒有系統提醒「哪些案件已過保存年限可銷毀」「哪些當事人物品還沒返還」
- 銷毀要符合保密（不能隨便丟，含當事人隱私 / 機密）

**現在怎麼做**
- 紙本封箱進倉、Excel 結案清單、電子檔歸入結案資料夾。保存年限多靠人工 / 沒在管。
- **架構意涵**：結案是 matter lifecycle 的終態，要觸發：(1) 帳務結清檢核（代收代付是否結清、後酬是否該收）、(2) 卷宗歸檔狀態、(3) 設定保存到期日 → 又是一個 **time-driven 提醒**（保存年限到期 = cron 掃 `pending_deadlines` 的另一個 deadline 類型，但用「年」尺度）。對映既有 `snapshot` / 歸檔概念。

## 架構總結（給 legal-admin 設計用）

**可直接沿用既有 company-ops 模組（改名 / 微調）**
- `customers` → `clients`（自然人 / 法人都有先例）+ 新增「對造」角色
- `attachments` → 卷宗 / 書狀（加文件類型 + 審級 + 版本）
- `transactions` → 律師費帳務 + **分帳：自有 ledger / 代收代付 ledger**
- `tasks` → 案件待辦
- `approvals` HITL gate → 利衝檢查 gate、後酬倫理檢核 gate
- 知識庫 + 機密軸 → 法律見解 / 案件策略（高度機密，預設 confidential）

**律所獨有、需新建**
- `matters`（案件，含審級子紀錄：案號 / 股別 / 法院隨審級變）
- `conflict_check` tool（接案前強制 gate）
- 計時計費（time entries + 帳單生成 + 多收費模式）
- **`pending_deadlines` + 期間計算引擎**（time-driven，複製 escalation 三層投遞架構）

**最重要的架構結論（印證老闆主張）**
律所最痛的時限管理是 **time-driven**，與既有 escalation 的 `pending_escalations` 笨投遞模式同構。應把它做成 **system 層服務**：`pending_deadlines` 表 + OS cron 保證層（每日掃多閾值倒數）+ claude -p 品質層（產脈絡提醒）+ in-session 即時層。時限在律師沒開 Claude 時照樣倒數、照樣提醒 —這是本 vertical 相對「網頁 AI / 通用助理」不可取代的核心價值。期間計算引擎必須內建「翌日起算 + 假日順延 + 在途期間 + 內部緩衝（驗證了老闆的『19 天 = 20 − 1』）」，且每次回傳**附法條依據**（法律天數不可憑印象，呼應反捏造原則）。

## 0. 一個必須先釘死的觀念：兩條獨立的時間軸

時限引擎處理的不是「一個倒數」，而是兩段串接：

```
[送達] ──(送達生效規則)──> [生效日] ──(民法§120 翌日起算)──> [起算日] ──(法定期間 + 在途期間)──> [理論末日] ──(民法§122 末日順延)──> [最終到期日]
```

- 「送達生效」與「期間起算」是**兩個不同的事件**，中間可能隔很久（例：寄存送達多 10 天、公示送達多 20 天）。引擎最常算錯的就是把「收到的那天」當起算點，忽略送達生效規則。
- 安全緩衝（老闆查到的「19 天」）是疊在最後的內部期限，**不可與法定 20 日混淆**——法定 20 日是不可退讓的硬牆，19 天只是事務所自訂的提早線。

---

## 1. 起算規則：始日不算入、翌日起算（民法 §120）

**民法第 120 條（期間之起算）**（逐字查證）：
> 「以時定期間者，即時起算。**以日、星期、月或年定期間者，其始日不算入。**」

實務白話：以「日」計的法定期間，**送達生效當日（始日）不算**，從**次日（翌日）**為第 1 日起算。

- 例（民訴 §440 上訴期間，逐字查證）：「提起上訴，應於第一審判決送達後**二十日之不變期間**內為之。」司法院明確：「上訴期間自收受判決之翌日起算 20 日。」
- 例（民事抗告）：10 日不變期間，自「裁定送達**翌日**」起算。

> 引擎規則 R1：`起算日 = 送達生效日 + 1 天`（始日不算入）。所有「以日定」的期間一律如此，沒有例外。

---

## 2. 末日順延（民法 §122）— 只有末日適用

**民法第 122 條（期間終止之延長）**（逐字查證）：
> 「於一定期日或期間內，應為意思表示或給付者，**其期日或其期間之末日，為星期日、紀念日或其他休息日時，以其休息日之次日代之。**」

配合 **民法第 121 條**（期間之終止）：以日定者，以末日之終止為期間終止。

關鍵約束（第 4 點，極重要、引擎最易錯）：
- **只有「末日」遇假日才順延**。期間「中間」的星期六/日/國定假日**照常計入、不停止計算**（期間連續計算）。
- 末日若遇 週六 / 週日 / 國定假日 / 其他休息日 → **順延至次一上班日**（若次日仍是假日，繼續往後順延到第一個上班日）。

> 引擎規則 R2（中間假日）：理論末日 = 起算日 + (法定期間 + 在途期間 − 1)，計算過程中**不跳過任何假日**。
> 引擎規則 R3（末日順延）：若理論末日落在非上班日，逐日往後推到第一個上班日 = 最終到期日。

---

## 3. 在途期間（民訴 §162 + 司法院標準）— 距離/地區分級加算

**民事訴訟法第 162 條（在途期間之扣除）**：
> 「當事人不在法院所在地住居者，計算法定期間，應扣除其在途之期間。**但有訴訟代理人住居法院所在地，得為期間內應為之訴訟行為者，不在此限。**」

機制要點（查證自司法院/律師事務所解說）：
- 「扣除在途期間」在實作上 = **延長期間**（把往返路程的時間還給當事人，等同期間加上 N 天）。
- **適用對象**：法定期間中的**不變期間**（上訴、抗告期間）+ 部分通常法定期間（如聲請回復原狀期間）。**裁定所定期間（審判長/法院裁定的期間）不適用在途期間。**
- **重大除外（必做的判斷）**：若**有訴訟代理人（律師）住居於法院所在地**且能在期間內為訴訟行為，**不扣除在途期間**（在途 = 0）。→ 律所自己當代理人、且律所在受訴法院所在地時，通常吃不到在途期間。引擎要有這個 flag。
- **以誰的所在地為準**：以「當事人或訴訟代理人**實際所在地**」對受訴法院判斷。

**司法院《法院訴訟當事人在途期間標準》結構**（查證自全國法規資料庫 pcode=B0010020）：

| 條 | 內容 |
|---|---|
| 第 1 條 | 授權依據：民訴 §162 第 2 項 + 刑訴相關條文 |
| 第 2 條 | **住居於受訴法院管轄區域「內」**：按地方法院別列天數（區域內距離分級） |
| 第 3 條 | **住居於管轄區域「外」**：按受訴法院區域天數 **+** 住居地所在地方法院區域天數合計；天數不同時取**較長**者 |
| 第 4 條 | 施行日（最近一次修正生效：民國 107 年 7 月 1 日） |

**實際範例天數**（查證；用於 seed 資料底、實作前須以官方表逐筆核對）：

| 區域 / 法院 | 在途期間 |
|---|---|
| 臺北市中心 / 臺南市中心 / 高雄市中心（管轄區內核心） | 不計（0 日） |
| 臺北/士林/新北 相互間部分情形 | 0 ~ 2 日 |
| 臺中市中心 | 1 日 |
| 一般近距離地院轄區 | 2 ~ 4 日 |
| 屏東 / 臺東較遠區 | 4 ~ 8 日 |
| 澎湖 | 10 ~ 15 日 |
| **金門地方法院** | **1 ~ 19 日** |
| 馬祖（連江） | 10 ~ 19 日 |
| 大陸地區 / 港澳 | **統一 37 日** |
| 國外—亞洲 | 37 日 |
| 國外—歐洲/北美/南美/大洋洲 | 44 日 |
| 國外—非洲/南極洲 | 72 日 |

> 老闆查到的「19 天」極可能對應**金門/馬祖在途期間上限（19 日）**，而非「20 日 − 1 安全緩衝」。兩種解釋引擎都要支援，但這條查證結果指向：19 是真實存在的官方在途上限值。實際個案要看「金門當事人 + 哪個受訴法院」的精確組合，不可用單一 19 寫死。

> 引擎規則 R4：在途天數來自查表（受訴法院 × 當事人住居地），**先判斷「是否有住法院所在地的訴訟代理人」**（有 → 在途 = 0）；無代理人除外時，加算查表天數。

## 4. 不變期間 vs 可伸長/縮短期間（民訴 §163）

**民事訴訟法第 163 條（期間之伸長或縮短）**：
> 「期間，如有重大理由，得伸長或縮短之。**但不變期間，不在此限。**伸長或縮短期間，由法院裁定；但期間係審判長所定者，由審判長裁定。」

| 類型 | 能否伸縮 | 適用在途期間 | 救濟（遲誤） | 引擎處理 |
|---|---|---|---|---|
| **不變期間**（法律明定「不變期間」字樣：上訴 20 日、抗告 10 日等） | 法院**不得**伸長或縮短 | **適用**（可扣在途） | 可聲請回復原狀 | 硬牆，紅色最高優先；末日順延 + 在途加算 |
| **通常法定期間**（如聲請回復原狀 10 日、證人請求日費期間） | 原則不可伸縮，但部分依§163 重大理由可裁定 | 部分適用 | 視規定 | 次級牆 |
| **裁定期間**（審判長/法院裁定所定，如補正期間） | **得**依§163 伸長/縮短（法院裁定） | **不適用在途** | 通常無回復原狀 | 可變期間，引擎標示「可向法院聲請延長」 |

> 引擎規則 R5：每個 deadline 帶 `period_type ∈ {不變期間, 通常法定期間, 裁定期間}`。只有「不變/通常法定」吃在途期間；「裁定期間」不吃在途、且可聲請伸長。

## 5. 回復原狀（遲誤不變期間的救濟）

**民事訴訟法第 164 條（回復原狀之聲請）**（查證）：
> 「當事人或代理人，**因天災或其他不可歸責於己之事由，遲誤不變期間者，於其原因消滅後十日內，得聲請回復原狀。**」
> 「**遲誤不變期間已逾一年者，不得聲請回復原狀。**」（絕對失權上限）

**民事訴訟法第 165 條（程序）**：聲請回復原狀，因遲誤上訴/抗告期間者，向為裁判之原法院為之；其他向期間內應為訴訟行為之法院為之。須**釋明遲誤原因及原因消滅之時點**，並**同時補行**期間內應為之訴訟行為（如同時補提上訴狀）。

**刑事訴訟法第 67 條**（查證，含 2023/04/18 三讀修正）：
> 非因過失遲誤上訴、抗告或聲請再審期間，或聲請撤銷/變更審判長、受命/受託法官、檢察官裁定之期間者，於其**原因消滅後十日內**得聲請回復原狀。允許代理者，代理人之過失視為本人之過失。
> （修法重點：回復原狀及準抗告期間由原 **5 日延長為 10 日**——引擎若有舊資料須更新。）

> 引擎規則 R6：deadline 一旦標記「逾期」，引擎不要直接判死，應掛出回復原狀路徑：`原因消滅後 10 日內 + 距遲誤未逾 1 年（民事）`，並提示須「同時補行訴訟行為」。這是事務所避免「漏期＝過失」演變成「完全無救濟」的最後安全網。

## 6. 送達方式對「起算」的影響（最易算錯的一段）

期間起算前必須先決定「送達何時生效」。三種特殊送達的生效時點：

### 6a. 寄存送達（民訴 §138，現行條文逐字查證）
> 第 2 項：「**寄存送達，自寄存之日起，經十日發生效力。**」（此為 110/2021 年修正新增的 10 日緩衝；舊法寄存即生效，引擎不可沿用舊規則）
> 第 3 項：寄存文書自寄存日起，寄存機關保存二個月。

- 生效時點：寄存日 **+10 日**（最高法院 94 年第 1 次庭長法官會議：10 日**自寄存日之翌日起算**）。
- **與行政程序法不同**：行政程序法的寄存送達「寄存即生效」、無 10 日緩衝。律所訴訟用的是**民訴版（+10 日）**。引擎要分流程別。

> 計算串接示例（上訴 20 日 + 寄存）：判決寄存日 → +10 日生效 → 生效翌日為起算第 1 日 → 連續算 20 日（中間假日不停）→ 末日遇假日順延。

### 6b. 公示送達（民訴 §152，逐字查證）
> 「公示送達，自將公告或通知書黏貼公告處之日起，公告於法院網站者，自公告之日起，其登載公報或新聞紙者，自最後登載之日起，**經二十日發生效力**；就應於外國為送達而為公示送達者，**經六十日發生效力**。」

- 境內生效：黏貼/網站公告日起 **+20 日**；應於外國者 **+60 日**。
- 引擎需記錄三種起算基準（黏貼處日 / 法院網站公告日 / 最後登報日），取適用者。

### 6c. 囑託送達
- 由受訴法院囑託他機關（他法院、外國機關、駐外館處、監所長官對在監人等）代為送達。**生效時點仍依「實際送達完成」或對應送達種類規則認定**，本身不另設固定加算日數；引擎處理上以「回證載明的送達完成日」為生效日，但對「在監所人」「外國送達」要特別預留人工確認旗標（送達回證回收常延遲）。

> 引擎規則 R7：每筆 deadline 帶 `service_type ∈ {一般送達, 寄存送達, 公示送達_境內, 公示送達_外國, 囑託送達}`，由此決定 `生效日`。一般送達 = 收受日當日生效；寄存 = +10；公示境內 = +20；公示外國 = +60；囑託 = 依回證完成日（標人工複核）。

```
  party_location,           // 當事人住居地（區域代碼，用於在途查表）
  court,                    // 受訴法院（區域代碼）
  has_local_agent,          // bool：是否有訴訟代理人住居法院所在地且能於期間內行為（§162 但書）
  safety_buffer_days,       // 事務所內部安全緩衝（例：1，產生 19 天內部線）
}
```

### 計算步驟（依序，不可調換順序）
1. **決定送達生效日** `effective_date`（R7）：
   - 一般送達 → `service_base_date`
   - 寄存送達 → `service_base_date + 10`（自寄存翌日起算 10 日）
   - 公示送達_境內 → `service_base_date + 20`；公示送達_外國 → `service_base_date + 60`
   - 囑託送達 → 回證載明完成日（標 `needs_manual_review=true`）
2. **決定起算日** `start_date = effective_date + 1`（R1，始日不算入）。
3. **計算在途天數** `transit_days`（R4）：
   - 若 `period_type == '裁定期間'` → `transit_days = 0`（裁定期間不適用在途）
   - 否則若 `has_local_agent == true` → `transit_days = 0`（§162 但書）
   - 否則查《在途期間標準》表：`transit_days = lookup(court, party_location)`（區域外 = 受訴法院區域 + 住居地區域，取較長）
4. **計算理論末日** `nominal_due = start_date + (statutory_days + transit_days) - 1`（R2，中間假日全計入、不跳過）。
5. **末日順延** `final_due`（R3）：
   - while `is_holiday(final_due)`（週六/週日/國定假日/休息日）→ `final_due += 1` 直到上班日。
   - （注意：只對末日做，步驟 4 的中間日**不**做。）
6. **計算事務所內部期限** `internal_due = final_due - safety_buffer_days`（再對 internal_due 也做一次「往前不落在假日」的提醒對齊，或直接以上班日呈現）。
7. **產出救濟備援**（R6）：附「若遲誤」的回復原狀視窗：民事 = 原因消滅後 10 日內且距遲誤未逾 1 年；刑事 = 原因消滅後 10 日內。

### 輸出（Output schema）
```
{
  effective_date,           // 送達生效日（含特殊送達加算）
  start_date,               // 起算日（翌日）
  transit_days,             // 實際採用的在途天數 + 來源（查表 row / 但書歸零）
  nominal_due,              // 順延前理論末日
  final_due,                // 法定最終到期日（已末日順延）— 硬牆
  internal_due,             // 事務所內部期限（含安全緩衝）— 行動線
  days_remaining,           // 距 internal_due 的工作日 / 日曆日
  recovery_window,          // 逾期時的回復原狀條件與截止
  calc_trace: [             // 每步驟的可稽核軌跡（律師覆核用、避免黑箱）
     '送達生效=寄存日+10', '起算=生效+1', '在途=金門→受訴法院 N 日(查表 vX)', '末日順延：6/7(日)→6/8(一)'
  ],
  legal_basis: ['民法§120','民法§122','民訴§162','民訴§440','在途期間標準 107.7.1版'],
  needs_manual_review,      // 囑託送達/外國送達/在途表罕見組合 → 強制人工複核
}
```

**設計原則（給時限引擎、對齊 system-layer 架構）**：
- 引擎輸出**永遠同時給「法定末日（硬牆）」與「內部期限（行動線）」**，且 `calc_trace` 必可被律師逐步覆核——時限算錯是過失，引擎不能是黑箱。
- 凡碰到 `needs_manual_review`（囑託/外國/在途罕見組合）一律**不自動結案、推給人複核**——fail-toward-有人看，呼應既有 escalation「fail-toward-有人收」哲學。
- 此引擎天然適合跑在 **system 層**（OS cron 每日掃所有未結 deadline + claude -p 品質層產出每日到期清單 + in-session push 即時提醒），因為倒數與人是否開 Claude 無關——與既有 `flush_escalations.py` 三層投遞範式完全同構。

## 8. 需要的資料底（data layer）

### 8a. 台灣國定假日 / 辦公日曆（用於 `is_holiday()`）
- **權威來源**：行政院人事行政總處（DGPA）《政府行政機關辦公日曆表》。每年 6/30 前公告次年。授權各界自行下載。官網 `dgpa.gov.tw/informationlist?uid=41`。
- **政府開放資料**：政府資料開放平臺 data.gov.tw dataset 14718 / 26557（《政府行政機關辦公日曆表》），含「日期 / 星期 / 是否放假 / 說明」。
- **可直接用的 JSON 鏡像（社群維護、結構乾淨，建議當 cache，再以 DGPA 官方對賬）**：
  - GitHub `ruyut/TaiwanCalendar`：逐年 JSON，欄位 `date / week / isHoliday / description`。
  - GitHub `880831ian/taiwan-calendar`：附 API + 民國年/星期英文等擴充欄位。
- **重要**：辦公日曆已涵蓋「補班日 / 彈性放假調移」——這正是 §122 末日順延所需的「上班日」真值（不能只用週末 + 固定國定假日推算，會漏掉颱風假/調整放假/補班）。引擎的 `is_holiday()` 必須吃這份官方表、不可自行硬算。

### 8b. 在途期間表（用於 `lookup(court, party_location)`）
- **權威來源**：司法院《法院訴訟當事人在途期間標準》，全國法規資料庫 pcode=**B0010020**（民刑事用）；行政訴訟另有《行政法院訴訟當事人在途期間標準》pcode=A0020097，須分流。
- **結構（建模）**：
  - 維度 1：受訴法院（地院/高院分院別 + 管轄區域）
  - 維度 2：當事人住居地（同一法院別的「管轄區內」分級，或「管轄區外」=他法院區域）
  - 規則：區域內查第 2 條表；區域外 = 受訴法院區域天數 + 住居地地院區域天數，天數不同取較長（第 3 條）；大陸港澳統一 37、國外按洲別 37/44/72。
  - 資料表建議 schema：`transit_period(court_code, party_region_code, days, basis_version, note)`，外加 `version='107.7.1'` 以便標準改版時版控（trace 要記是哪版）。
- **無官方 open data / API**：此表只有法規條文形式，**必須由人工逐筆建表 + 律師覆核**，並以條文版本號鎖定。建表後納入引擎 `calc_trace` 引用，方便日後標準修正時回溯。

## 9. 給後續設計的關鍵提醒（避免實作踩雷）

1. **始日 vs 末日不對稱**：始日不算（§120）、中間假日全算、只有末日順延（§122）——三者規則不同，是最高頻 bug 源。
2. **送達生效 ≠ 收件**：寄存 +10、公示 +20/+60 必須先算，否則整條時間軸位移。民訴寄存版（+10）與行政程序法版（即生效）不可混用。
3. **在途但書**：有「住法院所在地的訴訟代理人」→ 在途歸零；裁定期間不吃在途。律所自辦案件常落在「在途 = 0」。
4. **不變期間是硬牆**：法院不得伸縮（§163 但書），引擎不可給「申請延長」選項；裁定期間才可。
5. **19 天的真相**：查證指向金門/馬祖在途上限本就是 19 日（官方值），須與「20−1 安全緩衝」兩種來源分清；引擎用查表值、不寫死。
6. **回復原狀是最後安全網**：逾期不等於結束，10 日 / 1 年視窗要主動掛出。
7. **辦公日曆必用官方表**：補班/調移/颱風假無法自行推算，`is_holiday()` 依 DGPA 表。

## 0. 先把「19 天」這件事釘死（法定期間查證）

老闆查到的「19 天」假說成立：法定上訴期間是 **20 日**，事務所內部抓「**20 − 1 = 19 天安全緩衝**」是合理的內控做法。查證如下：

| 期間類型 | 法定天數 | 法條依據 | 起算 |
|---|---|---|---|
| 民事上訴（一審→二審） | **20 日**不變期間 | 民事訴訟法 §440 | 判決正本**送達翌日**起算 |
| 民事上訴第三審 | **20 日**不變期間 | 民訴 §481 準用 §440；第三審**律師強制代理** | 二審判決送達後 |
| 民事上訴狀未表明理由者，補提理由書 | **20 日** | 民訴 §471 | 提起上訴後 |
| 刑事上訴 | **20 日** | 刑事訴訟法 §349 | 判決送達後 |
| 民事抗告 | **10 日**不變期間 | 民訴 §487 | 裁定送達後 |
| 刑事抗告 | **5 日** | 刑訴 §406 | 裁定送達後 |

關鍵實務細節（軟體必須處理對，否則算錯=執業過失）：

1. **不變期間不可伸縮**（民訴 §163），法院不得延長/縮短。漏掉只能聲請「回復原狀」（原因消滅後 10 日內），但屬例外救濟、不能當常態。
2. **連續計算、不扣假日**；但**末日若為週六/日/國定假日，順延至次一上班日**。
3. **在途期間（在途天數）會加計**——但有陷阱：若**有住法院所在地、且有特別委任的訴訟代理人（律師）**，則**不扣在途期間**；其餘情形依當事人所在地加計。這是軟體最容易算錯的地方，台灣本土工具的核心價值就在這。
4. **送達日**才是起算錨點——軟體必須以「判決/裁定送達日」為輸入，而非「判決日」或「收到通知日」。
5. 司法院有官方**期間試算工具**（gdgt.judicial.gov.tw），已含在途期間、末日順延參數。**這代表「日期試算」本身不是護城河、司法院免費就有；護城河在「把每個案件的送達日抓進來、自動倒數、主動提醒、且人沒開電腦也在跑」。**

> 設計啟示：助理不應自己手算複雜的在途期間並對外宣稱權威，而應 (a) 以送達日為輸入、(b) 預設用「20 日 − 安全緩衝」產生內部死線、(c) 對在途/特別委任這類分歧情境**標記不確定、要律師確認**，而非靜默算死。這正好契合本系統的反捏造原則。

## 1. 台灣本土工具

台灣市場分兩類玩家，定位完全不同：**(A) 法學/AI 內容工具**（Lawsnote、LawChat 等，幫你「查法律 + 寫書狀」），與 **(B) 事務所管理系統**（康瓏、文易管、世盈、ezip 等，幫你「管案件 + 管期限 + 計費」）。**沒有一家把兩者 + LINE 主動推播完整縫起來**，這是切入點的根源。

### A 類：法學搜尋 / AI 書狀（不是案件管理）

**Lawsnote 七法**
- 核心：法學資料庫搜尋引擎（判決/法規/函釋）+ AI 法律問答（RAG、附來源連結、宣稱去幻覺）。
- 2025/11 推 **AI 書狀生成**：上傳卷證→一鍵生起訴狀/意見書，3 分鐘出可下載 Word，宣稱 RAG 綁自家資料庫不產假案。另有 DocPie（契約自動生成+審閱）、「法務掌」(一站式法律服務平台)。
- 價位：年費 **NT$7,800/人**（單買）；3 人團每人 **NT$5,883/年**。
- **時限管理：無**。這是搜尋/生成工具，不管案件死線。
- **計費：無**（不是事務所管理系統）。
- 優：法學內容深、AI 書狀是台灣最成熟的。缺：完全不碰案件/期限/行政管理。
- 提醒：兩名創辦人因爬法源資料被判刑（爬蟲判例），品牌有法律爭議史，做法律業 vertical 要知道。

**LawChat / Lawbot / 1890 線上書狀**：同屬 AI 書狀生成賽道（「10 分鐘完成 2 小時書狀」「催討帳款線上生成」），都是內容生成、非管理系統。

### B 類：事務所管理系統（本研究真正的競品）

**康瓏多媒體（7coins）** — 功能最完整的本土代表
- 23 個模組：基本資料、我的工作平台、客戶、案件進度、案件帳務、**庭期管理**、文件、會計、知識庫、通知、休假、**利益衝突查核**、Google 日曆匯入、**書狀追蹤**、行動版等。
- **時限做法**：核心賣點是「對不得延誤之工作**強制提醒直至回報完成**」，工作分類含**末日 / 庭期 / 撰狀 / 發文 / 收文 / 法顧合約到期**；可預訂後續所有提醒、追蹤至完成、可轉發給同事。
- **計費**：四種模式——**計時制（Hourly Rate，勾選工作即算工時/金額）**、專案制、包案制、顧問諮詢；全欄位可調時數/時薪。
- **通知**：工作平台 + 內部公告 + 交辦 + Hotline 專線 + 信箱。**沒有 LINE**。
- 價位：**官網不公開、需電洽**（典型台灣 B2B 軟體賣法）。
- 優：模組齊、本土法律流程貼合（末日/書狀/利益衝突）。缺：UI 偏傳統、提醒是「登入才看得到的工作平台」式（被動拉、非主動推）、無 LINE、無公開價、無 AI。

**文易管 / E 律師事務所管理系統**：案件進度追蹤、結案統計、律師案量、信封/發票列印、**收款提醒**、工時成本統計、客製績效報表；可線上/離線/雲端混用。定位中小所、彈性部署。同樣電洽報價、無 LINE、無 AI。

**世盈科技、ezip（法律期限與工時管理系統）、益盛科技**：同質競品。ezip 主打「法律期限 + 工時」；益盛把「案件狀態/任務/逾期款/客戶」整合一平台 + 收款提醒。皆傳統管理系統路數。

**本土小結**：B 類功能上「期限提醒 + 計時/計費」是**標配（table-stakes）**，但全部是 **(1) 被動式（要登入工作平台/開系統才看得到）、(2) 無 LINE 主動推播、(3) 無 AI、(4) 不公開價、需業務接觸、(5) 安裝/導入重**。A 類有 AI 但完全不管期限與行政。**「AI + LINE 主動推 + 自動定時倒數」這個組合，台灣目前是空白。**

## 2. 國際工具

國際成熟度遠高於台灣，尤其在 **court rules engine（法院規則引擎）+ AI 抽期限** 上。但它們的「規則引擎」綁的是美國法院規則，**無法直接搬到台灣**——這是台灣本土仍有空間的結構性原因。

**Clio**（市場龍頭，2025/11 估值 USD 50 億，以 USD 10 億併 vLex）
- **時限/docketing（旗艦能力）**：
  - **Court Rules 引擎**：依「數百個美國法院的規則」自動產生事件與死線（不用人手算日期）。
  - **Manage AI / Clio Duo**：**從法院文件自動抽出死線→建成已驗證的行事曆事件與任務**（court document → verified calendar events），這是它最強的賣點：把「漏期限」風險用 AI 從文件源頭堵住。
  - 第三方整合（如 CourtDrive）自動同步 ECF/eNotice 法院文件。
- **AI**：Manage AI（前身 Clio Duo）——案件摘要、文件分析、死線抽取、客戶溝通草擬、任務排序、帳單草擬、收據對帳。另有獨立深度法律研究線（Harvey 屬不同產品線、Clio Duo 偏 practice management 內嵌）。
- **計費**：完整 time tracking + billing + trust accounting。
- 價位：EasyStart **$49/月**（單人）、Essentials **$89/月**、Advanced/Expand **$119–169/月**（中型所）。**per-user 訂閱、價格透明**。

**MyCase**
- **時限**：單一集中行事曆 + **court rules-based events**、整合簡訊與**事件簡訊提醒（text reminders）**、與 Outlook/Gmail/O365/iCal 同步。**MyCase 已內建 SMS 提醒，這點最接近「主動推」**。
- **AI（MyCase IQ）**：潤稿（email/筆記/摘要）、**翻譯助理**（跨語言保留法律語氣）。AI 偏「寫作/溝通」，期限抽取不如 Clio 強。
- 價位：Basic **$39**、Pro **$79**、Advanced **$99**（per user/月）。

**PracticePanther**
- **時限**：**court rules-based events**，自動算死線、消除手算日期；行事曆同步、自動提醒、即時通知；原生電子簽名。
- **AI**：有提及 AI 能力但公開資料較少、不是其主打。
- 定位：workflow 自動化 + billing + client comms，中小所友善。

**國際小結**：三家都把 **court rules engine + 自動算死線**當標配，Clio 再用 **AI 從法院文件抽死線**拉開差距。**關鍵洞察：它們的「規則引擎」是美國法院 hard-code 的、台灣用不了；台灣的法定期間（20/10/5 日 + 在途 + 末日順延）相對單純，反而是「用文件送達日驅動 + AI 抽送達日 + LINE 推」就能做到八成價值。** 另外國際工具**全部是 web/app 被動式**，沒有一家是「人沒開系統、system-layer 仍在替你倒數並主動戳你 LINE」。

## 3. 台灣律師收費規範與行情

**規範性質（重要）**：台灣**沒有強制的全國統一收費上限**。現行是**律師公會的「建議性」收費標準**（如台北律師公會 83.9.9 會員大會通過的酬金收受辦法，沿用至今）+ **律師倫理規範的硬規定**。換言之，上限數字多屬「宜（建議）」而非「不得逾（強制）」，但有兩類是硬性的。

**硬性規定（律師倫理規範 §35）**：
- 律師**必須向委任人明示酬金數額或計算方法**（→ 軟體應內建報價/委任書的費用條款，可成 compliance 賣點）。
- **家事案件、少年事件、刑事案件不得約定後酬（成功報酬）**。後酬僅限民事財產權訴訟（常見於工程款等）。

**台北律師公會建議標準（"宜"，非強制）**：
| 項目 | 建議金額 |
|---|---|
| 討論案情（鐘點） | 每小時 NT$8,000 以下 |
| 撰擬函件 | NT$20,000 以下 |
| 意見書等文件 | 每件 NT$80,000 以下 |
| 出庭費 | 每次 NT$20,000 以下 |
| 各審書狀 | 每件 NT$50,000 以下 |
| **時間計費（標準）** | **每小時 NT$8,000 以下；案情複雜得增至 NT$12,000** |
| 民事案件每審酬金總額 | 宜 NT$50 萬以下；標的 ≥500 萬者得增加，但增加額每審不宜逾標的 3% |

**法定/法院核定（這部分是硬的）**：
- **法院選任律師及第三審律師酬金核定支給標準**（法規 B0010037）：法院斟酌案情繁簡、結果、勤惰核定；民事財產權訴訟於標的金額/價額 **3% 以下、最高不得逾 NT$50 萬**。

**市場實際行情（2026）**：
- 諮詢費：1 小時約 **NT$2,000–8,000**。
- 書狀撰寫：約 **NT$10,000–50,000**。
- 常見計費結構：**按審級包案**（最普遍）、**鐘點制**、**單件式**（諮詢/書狀/出庭分開計）、**後酬制**（限民事財產權）。

> 設計啟示：助理的**計費模組必須支援台灣特有結構**——「按審級」「鐘點」「單件式」「後酬（僅限民事且需 compliance 警示）」。並可內建 §35 合規護欄：建立委任/報價時提醒「需明示酬金」、偵測家事/刑事/少年案件若設後酬則**主動擋下並上報**（完美對應現有 escalation 機制）。

## 4. 總結：差異化切入點

### 4.1 Table-stakes（不做就不及格，必做）
1. **法定期間死線管理**：以**送達日**為錨、自動產生「20/10/5 日 − 安全緩衝」內部死線、末日順延、在途期間至少標記提醒。**這是律所軟體的命脈、漏=執業過失。**
2. **庭期管理 + 提醒**。
3. **案件進度追蹤**（當事人、案號、審級、承辦律師）。
4. **書狀管理/追蹤**（撰狀、收發文）。
5. **計時/計費**（鐘點 + 按審級 + 單件式 + 後酬，且符合台灣行情與 §35）。
6. **利益衝突查核**（本土系統都有、客戶會問）。
7. **資料安全/可信**——客戶隱私是律所紅線（正好對應本系統的 floor 兩道牆）。

### 4.2 現有工具的明顯缺口（= 我們的切入點）
1. **被動 vs 主動，且時間驅動**：本土 B 類是「登入工作平台才看到提醒」、國際是「開 web/app 才看到」。**沒有一家是 system-layer 時間驅動、人沒開系統也在替你倒數並主動戳你**。死線是時間驅動的——這正是老闆主張的核心，也是現成 escalation 基建（OS cron `flush_escalations.py` 保證層 + `claude -p` 品質層 + in-session push 即時層）**天生契合**的場景。律所版可把「pending_escalations」概念換成「pending_deadlines」：每筆案件死線進 DB、cron 每天掃、到緩衝門檻就主動 LINE 推承辦律師（T-19/T-10/T-3/T-1 多段提醒、且**自動 retry / 升級到合夥人**——漏期限=過失，必須 fail-toward-有人收，這正是 escalation 收件人 coalesce 邏輯）。
2. **無 LINE**：台灣律師與助理的日常溝通在 LINE。本土系統零 LINE、國際只有 MyCase 有 SMS。**「死線/庭期/客戶訊息直接進 LINE、回覆即操作」是台灣在地化降維打擊**，且本系統的 line-channel 基建已成熟。
3. **AI 與管理脫節**：Lawsnote 有 AI 書狀但不管案件；管理系統有期限但無 AI。**「AI 抽送達日/期限 + AI 草擬書狀/答辯 + 自動排進死線倒數」一條龍**，台灣無人做（這正是 Clio Manage AI 在美國做的，台灣空白）。
4. **價格不透明 + 導入重**：本土全部電洽、需業務、安裝重。**透明、輕量、LINE 即開即用、訂閱制**對單一律所友善。
5. **合規護欄當賣點**：§35「明示酬金」「家事/刑事/少年不得後酬」可做成自動檢查 + 上報，現有 HITL gate + escalation 直接複用。

### 4.3 一句話定位
> **「跑在 LINE 上、人沒開電腦也在替你倒數法定死線的 AI 律所行政助理」**——把國際工具的 court-rules-deadline 自動化 + Clio 的 AI 抽期限，用**台灣法定期間（單純但漏不得）+ LINE 主動推 + system-layer cron 時間驅動**重組，補上本土管理系統「被動、無 LINE、無 AI」與本土 AI 工具「不管案件死線」之間的整片空白。死線管理是天生的 system-layer 工作，而現有 escalation 三層投遞基建（cron 保證層 / claude -p 品質層 / in-session 即時層）就是現成範本，把 `pending_escalations` 換成 `pending_deadlines` 即可。

### 4.4 給 legal-admin branch 的具體建議（架構落點）
- **deadline 引擎**：新增 `deadlines` 表（案件/類型/送達日/法定天數/緩衝/末日順延/在途旗標/承辦律師 line_id/狀態），用既有 `pending_escalations` 同款 in-tx enqueue + OS cron 笨投遞器掃描 + 多段提醒（T-19/T-10/T-3/T-1）+ 漏報升級合夥人（沿用收件人 coalesce）。
- **反捏造對齊法律紀律**：在途/特別委任分歧情境**標記不確定要律師確認**，不靜默算死；日期試算可呼叫/比對司法院官方工具，不自稱權威。
- **計費**支援台灣四結構 + §35 合規 gate（後酬偵測家事/刑事/少年→擋+上報）。
- **單一律所、不做多事業體**：可移除 `business_unit` 維度，floor 改成「合夥律師 / 受僱律師 / 行政助理 / 機密(boss)」這類律所角色層，沿用兩道牆做客戶機密隔離（律師業對 client confidentiality 敏感度極高，floor 模型反而更好賣）。

## 2. 律所需要、現在完全沒有的新模組 / 資料表

以下六個是 SME-AI-Kit 完全沒有、律所必備的。資料模型概要（沿用既有慣例：新表只走 migration、不寫進 schema.sql；ISO8601 localtime；FK ON DELETE 策略沿用既有保留稽核的做法）。

### 2.1 `matters`（案件）— orders 的繼任者，但語意全換
案件是律所的核心聚合根，幾乎所有其他表都掛在它下面。
```
matters(
  id, matter_no TEXT UNIQUE,         -- 案號（事務所內部編號，如 2026-民-001）
  title TEXT,                         -- 案由（如「○○給付買賣價金事件」）
  client_party_id  → parties(id),    -- 委任人
  practice_area TEXT,                -- 民事/刑事/行政/家事/智財/勞資/非訟
  court TEXT,                         -- 繫屬法院（算在途期間用）
  court_case_no TEXT,                -- 法院案號（如 112年度訴字第XXX號）
  stage TEXT,                         -- 一審/二審/三審/執行/偵查/調解/結案
  status TEXT,                        -- open / on_hold / closed / archived
  fee_type TEXT,                     -- hourly 時薪 / fixed 包案 / contingency 成功報酬 / retainer 顧問
  fee_terms TEXT,                    -- JSON：時薪率、包案總額、成功報酬%
  lead_attorney TEXT,               -- 主辦律師
  opened_at, closed_at,
  confidential INTEGER DEFAULT 0,    -- 沿用知識機密軸概念，控 floor 可見
  created_at
)
```
改法重點見 §3.1。

### 2.2 `deadlines`（時限）— 全新、本 vertical 的命根
**老闆的核心主張落點：時限是時間驅動的、人沒開 Claude 也在倒數，所以必須起在 system 層（cron）。** 這張表 + cron flusher 是整個 vertical 的差異化價值。
```
deadlines(
  id, matter_id → matters(id) ON DELETE CASCADE,
  type TEXT,                         -- appeal_civil / appeal_criminal / petition_訴願 / admin_litigation / answer 答辯 / brief 準備書狀 / statute_of_limitation 消滅時效 / custom
  description TEXT,                  -- 「對一審判決提起上訴」
  -- 法定期間計算三件套（不可省，算錯=執業過失）
  trigger_event TEXT,               -- 起算事件（如「判決送達」）
  trigger_date TEXT,                -- 起算日（送達日）YYYY-MM-DD
  statutory_days INTEGER,           -- 法定日數（如民事上訴=20）
  statutory_basis TEXT,             -- 法條依據（如「民訴§440」）— 反捏造、強制附依據
  in_transit_days INTEGER DEFAULT 0,-- 在途期間加計（依當事人所在地，可為 0）
  statutory_deadline TEXT,          -- 法定末日（trigger 次日起算 + statutory_days + 在途，遇假日順延）
  buffer_days INTEGER DEFAULT 1,    -- 內部安全緩衝（老闆查到的「19 天」= 20−1）
  internal_deadline TEXT,           -- 內部期限 = statutory_deadline − buffer_days（實際盯的就是這個）
  status TEXT,                       -- pending / filed 已遞交 / extended 已聲請展延 / missed 已逾期 / cancelled
  assignee TEXT,                    -- 負責律師
  escalation_lead_days TEXT,        -- JSON：提前幾天上報（如 [7,3,1,0]），餵 cron
  filed_at, filed_by,
  created_at
)
```
**設計要點**：
- 起算採「次日起算」（民法§120Ⅱ「始日不算入」），末日遇例假日順延（民訴§161 準用民法§122）——這些計算規則必須寫死在 service 層，不可讓 LLM 即時心算。
- `statutory_basis` 強制非空，比照 knowledge 的 `source_quote` 反捏造原則：每個法定天數都要有法條依據可稽核。
- `escalation_lead_days` 是 cron 投遞器的輸入：cron 每天掃 `internal_deadline − today ∈ escalation_lead_days` 的列，enqueue 進既有 `pending_escalations`，完全復用三層投遞（cron 保證層 + claude -p 品質層 + in-session 即時層）。**這正是老闆「services 起在 system 層」主張的最佳載體。**

### 2.3 `conflict_checks`（利益衝突檢查）— 全新、收案前強制關卡
律師接案前依《律師倫理規範》必須做利益衝突檢查（不能同時代理對造、不能對抗舊當事人）。
```
conflict_checks(
  id, matter_id → matters(id),       -- 觸發此檢查的（候選）案件，收案前可先 NULL
  query_name TEXT,                   -- 被查名字（潛在當事人/對造）
  query_id_no TEXT,                  -- 身分證/統編（精準比對）
  result TEXT,                       -- clear 無衝突 / potential 疑似 / blocked 有衝突禁接
  hits TEXT,                         -- JSON：命中的既有 party_id / matter_id 清單
  resolution TEXT,                   -- 如何處理（拒接 / 設 ethical wall / 當事人同意書）
  checked_by, checked_at,
  approved_id → approvals(id),       -- potential 以上走 HITL，合夥人簽核才可接
  created_at
)
```
**設計要點**：跨比對 `parties`（所有歷史當事人+對造）。`blocked` 結果應硬接線觸發 escalation 給合夥人（沿用 #173 的 in-tx enqueue 模式）。利衝隔離牆落地用既有 `access_zones`/`grants`。

### 2.4 `time_entries`（計時）— 全新、時薪計費命脈
時薪制律所的營收基礎；包案制也需記錄成本。
```
time_entries(
  id, matter_id → matters(id) ON DELETE CASCADE,
  attorney TEXT,                     -- 計時律師（對應 employees）
  work_date TEXT,
  minutes INTEGER,                   -- 以分鐘記（律所慣例 6 分鐘=0.1 小時為單位）
  description TEXT,                   -- 工作內容（出帳時會列給當事人看）
  billable INTEGER DEFAULT 1,        -- 可計費 / 不可計費
  rate REAL,                         -- 該筆適用時薪（可從 matter.fee_terms 帶入）
  billed INTEGER DEFAULT 0,          -- 是否已開帳單
  invoice_ref TEXT,
  created_at
)
```
**設計要點**：可借鏡 leave 的「時數扣減」與 transactions 的關聯模式。出帳時彙總成一筆 income transaction + 信託帳扣抵。

### 2.5 `trust_ledger`（信託/受託款帳）— 全新、合規紅線
律師受託保管當事人款項（預付律師費、和解金、代收款）依法**必須與事務所自有資金嚴格分離**（信託專戶），這是律師懲戒的高風險區。不可塞進 `transactions`。
```
trust_ledger(
  id, matter_id → matters(id),
  party_id → parties(id),            -- 該款項屬哪個當事人
  entry_type TEXT,                   -- deposit 存入 / disbursement 動支 / transfer_to_fee 轉律師費 / refund 退款
  amount REAL,
  balance_after REAL,                -- 該當事人信託餘額（每筆後結餘，便於對帳）
  purpose TEXT,
  bank_ref TEXT,                     -- 信託專戶交易憑證
  approved_id → approvals(id),       -- 動支一律走 HITL gate（resume_params 鎖金額+收款人）
  recorded_by, entry_date,
  created_at
)
```
**設計要點**：
- 動支（disbursement / transfer）**強制走既有 approvals gate**——gate 的「金額/收款人一字不差比對 + 單次消費 consumed_at」正是防止信託款被誤動支或重複動支的理想機制（比一般記帳更需要）。
- 每個 party 的信託餘額不可為負（CHECK），且事務所層級「信託專戶總額 = Σ各當事人信託餘額」要可對帳。
- 任何動支硬接線 escalation 給合夥人（沿用 in-tx enqueue），合規留底落 `interaction_log`。

### 2.6 `documents`（書狀/文件版本）— 全新（attachments 升級）
attachments 只存「一個路徑指標」，律所需要文件分類 + 版本 + 保密等級 + 與時限/案件關聯。
```
documents(
  id, matter_id → matters(id) ON DELETE CASCADE,
  doc_type TEXT,                     -- pleading 書狀 / contract 契約 / evidence 證物 / court_doc 法院文書(判決/裁定/通知) / correspondence 往來 / kyc 委任資料
  title TEXT,
  version INTEGER DEFAULT 1,
  status TEXT,                        -- draft / review 待審 / final 定稿 / filed 已遞交
  confidential INTEGER DEFAULT 0,    -- 沿用機密軸 → floor 可見度
  deadline_id → deadlines(id),       -- 此書狀對應哪個時限（遞交即關閉時限）
  approved_id → approvals(id),       -- 對外/向法院遞交前走審核
  file_path TEXT,                    -- 沿用 attachments「存路徑不存檔」
  authored_by, created_at
)
```
**設計要點**：底層仍可復用 `attachments` 表（target_type='document'），或將 attachments 升級為其物理載體。「向法院遞交」這個動作應同時：更新 document.status='filed' → 關閉對應 deadline → 取消該 deadline 的 escalation。

## 3. 「改造」項的具體改法方向

### 3.1 orders → matters
- **重命名表 + 模組**：`orders/` → `matters/`，`orders` 表 → `matters`。
- **砍掉物流/QC/庫存全鏈**：移除欄位 `qc_status / qc_notes / qc_checked_* / driver / estimated_delivery / delivered_at / items / discount_applied`；移除工具 `qc_order / fulfill_order`（連帶移除其扣庫存邏輯，因 inventory 整個拔）。
- **狀態機改寫**：`pending→confirmed→shipped→delivered→paid` 改成 **`open → on_hold → closed → archived`** 的案件生命週期。`cancel_order`（含已出貨上報 `order_cancelled_shipped`）→ 改成「結案/撤回委任」，escalation trigger 改名。
- **超門檻自建審核（#26）邏輯保留**：原本 create_order 超門檻 in-tx 自建 approval 的 `create_in_tx` 模式，平移成「收費條件超門檻 / 成功報酬案」自建審核——這套硬接線 + resume_params 鎖定的基礎建設直接復用。
- **新增關聯**：matter 成為 deadlines / time_entries / documents / trust_ledger 的父鍵。

### 3.2 customers → parties（當事人 + 對造 + 利衝）
- **`type` 值域改寫**（CHECK constraint）：`customer/supplier/distributor` → `client / opposing / opposing_counsel / third_party / court`。
- **砍銷售聚合欄**：`total_ordered / total_fulfilled / total_paid / last_*_date / discount_rate` 這些電商語意全移除；改放 **`id_no`（身分證/統編，利衝精準比對用）、`role_in_matter`、關聯的 matter_id**。
- **新增利衝索引**：name + id_no 建索引，供 `conflict_checks` 快速掃描歷史對造。
- **find_customer → find_party**：搜尋時要能跨 client/opposing 兩面找（收案前查「這個對造名字我們以前代理過嗎」）。

### 3.3 accounting → 拆「營業帳」+「信託帳」
- **`transactions` 沿用當營業帳**：律師費收入、薪資、租金、規費代墊。`category` 值域改法務（legal_fee_income / salary / office_rent / court_fee / outsource）。`related_order_id` → `related_matter_id`。
- **信託款一律不進 transactions**：另立 `trust_ledger`（§2.5），兩本帳物理分離，符合信託專戶合規。
- **`approval_threshold` 改語意**：原「採購/退款門檻」→「信託動支門檻 + 收費審核門檻」。
- **`check_overdue` 沿用**：改盯「當事人欠繳律師費」而非 B2B 應收。

### 3.4 hr / employees → 律所角色
- `role` 值域：`boss/manager/staff` → `partner / associate / paralegal / admin`。`permissions` 的 admin/manager/basic 映射：partner=admin、associate=manager、paralegal/admin=basic。
- `external_partners` → 特約律師 / 委外（鑑定人、公證、翻譯社）。`role` 自由文字沿用。
- `update_employee` 的 admin-gate（#10）原樣保留——改律師權限仍是高風險動作。

### 3.5 floor-map → 律所安全層
`floor-map.json` 的四層改成律所信任邊界（仍走「通用底 + onboarding 客製」原則，不寫死）：
- `confidential`（合夥人層）：看全部、含信託帳全貌、機密見解。
- `accounting` → `finance`（會計層）：financial_visibility=all，看營業帳+信託帳但不碰案件機密內容/HR。
- `general`（受僱律師/助理層）：看自己經辦案件，financial_visibility=none（看不到信託帳全貌）。
- `external`（行政/前台層）：收文、排程、不看案件實體內容。
- **利衝隔離牆**：對特定衝突案，用 `access_zones`+`grants` 在同層內再切——這是 floor（部門牆）之外的 case-level 隔離，現有兩道牆模型已足以承載。

| 時限類型 | 法定期間 | 法條依據 | 性質 |
|---|---|---|---|
| 民事上訴 | 判決送達後 **20 日** | 民事訴訟法 §440 | 不變期間 |
| 刑事上訴 | 判決送達後 **20 日**（已由原 10 日修正為 20 日） | 刑事訴訟法 §349 | 不變期間 |
| 訴願 | 行政處分達到/公告期滿次日起 **30 日** | 訴願法（§14） | 不變期間 |
| 行政訴訟（撤銷訴訟） | 訴願決定書送達後 **2 個月** | 行政訴訟法 | 不變期間 |
| 在途期間加計 | 當事人不在法院所在地時，依「法院訴訟當事人在途期間標準」加計（有特別代理人住法院所在地者不加計） | 法院訴訟當事人在途期間標準 | 加計項 |

## 1. 既有 system-layer 三層機制精確摘要

來源：`mcp-servers/business-db/shared/escalation.py`、`shared/db.py`、`flush_escalations.py`、`install.sh`、`CLAUDE.md`〈上報機制〉。

整套設計的關鍵思想是 **「業務寫入（in-tx enqueue 一筆 row）」與「投遞（讀 row → push）」徹底解耦**。寫入端在交易內把一筆待辦 row 落 DB（含已蓋章的收件人/來源/操作者），三條獨立投遞路徑各自從那張 row 推送。投遞器是「笨的」（dumb）——只讀 row → push → 更新 status，不重算任何身份或措辭。

### 1.1 寫入端（觸發 + 蓋章，`enqueue_escalation`）

- **觸發點**：業務 service（accounting / orders / hr / auth）在「真正執行那支 tool 的同一個 `with transaction()` 內」無條件呼叫 `enqueue_escalation(db, ...)`。agent 看不到、跳不過——與業務寫入同一原子 commit（`escalation.py:175-320`）。
- **蓋章（建立當下解析寫死）**：`actor`（經 `_resolve_trusted_actor`，floored session 取 verified `user_id`、忽略 agent 自填）、`target_line_user_id`（收件人 coalesce 鏈：`floor-map.escalation_target` 直接 user_id → `role=boss` → `permissions=admin` → `company.boss_line_id`）、`source_floor`（系統讀 `SME_FLOOR`、非 LLM 措辭）。投遞器**不重算**這些（`escalation.py:124-172, 224-255`）。
- **寫入 row 後**：呼叫 `request_escalation_flush()`（設旗標）+ `queue_session_injection(payload)`（排即時注入）。`db.py:55-64`。
- **commit 才觸發投遞**（`db.py:96-147`）：`transaction()` 在 commit 成功後 fire-and-forget 起品質層 + 即時層；**rollback 會清旗標**（業務沒寫入就絕不通知）。

### 1.2 三條投遞路徑

| 層 | 觸發 | 走訂閱 / metered | 防遞迴 / env | 角色 |
|---|---|---|---|---|
| **保證層（主）** OS cron `flush_escalations.py` | crontab `*/2 * * * *`（install.sh:191） | 純 Python urllib + LINE push API、不用 LLM、零成本 | `os.environ.pop("SME_FLOOR")` 不解析身份（flush_escalations.py:27） | 確定性、runtime 全關也跑 |
| **品質層** `claude -p` single-shot notifier（`spawn_notifier`） | business-db commit 後 fire-and-forget（db.py:133-134） | **走訂閱**：`env.pop("ANTHROPIC_API_KEY")`（escalation.py:759） | `SME_NOTIFIER=1` 防遞迴 + debounce lockfile 30s（escalation.py:683, 690） | 合併多筆、寫人話 |
| **即時層（best-effort）** in-session push（`inject_to_sessions`） | commit 後經 line-channel owner IPC socket 注入（db.py:139-145） | 不起新進程、純 socket | meta 必為 `Record<string,string>`（int/None 被 CC 靜默丟＝#182 根因，escalation.py:363-368） | idle 全權限 session 即時自醒 |

**保證層資料流**（flush_escalations.py + escalation.py:457-531）：cron 起 → `_load_tokens()` 從 `line-channels.json` / env / `.mcp.json` 三來源取 LINE token → `flush_pending_escalations(push_fn)` → SELECT `status='pending' AND target_line_user_id IS NOT NULL`（含線性 backoff `retry_count*base` 分鐘）→ 對每 row 原子 claim（`claimed_at` CAS）→ `push_fn(channel_id, to, text)` → 成功 `status='sent'` + 落 `interaction_log(action='escalation_sent')`；失敗 `retry_count+1`，達 `FLUSH_MAX_RETRY=5` 標 `failed` 終態。

**品質層安全細節**（`spawn_notifier`，escalation.py:666-787）：
- 走訂閱：`env.pop("ANTHROPIC_API_KEY")`（不走 metered API）。
- 全權限：`env.pop("SME_FLOOR")` + `pop("SME_FLOOR_MAP")`（才看得到 escalation row）。
- 防遞迴：`SME_NOTIFIER=1`，進程內不再起 notifier。
- 窄工具白名單：`--allowedTools ToolSearch mcp__line__reply mcp__line__reply_flex mcp__business-db__list_pending_escalations mcp__business-db__mark_escalation_sent`，`--disallowedTools Bash Edit Write Read WebSearch WebFetch Agent`，`--permission-mode dontAsk`。
- 極簡 mcp-config：只掛 `business-db` + `line`，寫到受保護 state dir（非 floor-writable `/tmp`）、`mkstemp` 唯一檔名 0600、>120s GC 舊檔。
- prompt（`_NOTIFIER_PROMPT`，escalation.py:625-645）硬約束：收件人「絕不自行更改或猜測」、不同 `target_line_user_id` 絕不合併（防跨收件人洩密）、做完即結束。

**防雙送投遞租約**（escalation.py:399-426）：cron 與 notifier 併發時送前先原子 claim（`_CLAIM_UPDATE`），只有搶到那路可送 + 落 log；`_CLAIM_TTL_MIN=10` 分後 reclaim。不變量 `_NOTIFIER_CLAIM_BATCH(8) × (LINE_PUSH_TIMEOUT_MS/1000=10s) << TTL(600s)`，test 有 cross-file guard 綁死。

**三層的設計分工就是律所框架的核心可複製模式**：保證層 = 確定性必達（律所最需要——漏期限＝過失）；品質層 = LLM 寫人話/合併；即時層 = 律師正開著 session 時即時自醒。

## 2. 一般化成律所「定時服務框架」

### 2.1 為什麼可以幾乎原樣複製

既有 escalation 是 **「事件驅動」**（業務 commit 觸發）。律所時限是 **「時間驅動」**（人沒開 Claude 也在倒數）。但兩者**投遞層完全相同**——都是「DB 有一張待辦 row → 三層投遞器推給人」。差別只在**寫入端的觸發源**：

- escalation 寫入端 = `enqueue_escalation`（業務 in-tx）。
- 律所時限寫入端 = **OS cron 每日掃 DB 算出今天/近期到期的時限 → enqueue 一批 row**。

所以律所框架 = 在既有「投遞三層」前面，**新增一條「cron 掃描 → enqueue」的時間驅動寫入端**。`pending_escalations` 表可直接泛化為通用 `pending_notifications`（加 `kind` 欄區分 `escalation` / `deadline_reminder` / `hearing_reminder` / `billing_reminder`），三層投遞器零改動就能推送律所提醒。

### 2.2 需要新增的 DB（律所領域表）

最小新增（沿用既有 `business_entities` / `pending_escalations` 風格）：

| 表 | 關鍵欄 | 用途 |
|---|---|---|
| `matters`（案件） | `id, client_id, matter_type(民事/刑事/行政), court, our_role, status, assigned_lawyer_id, opened_at` | 案件主檔 |
| `deadlines`（時限） | `id, matter_id, deadline_type(上訴/答辯/聲明/補正), legal_basis(法條), statutory_days, statutory_date, internal_date, trigger_event(判決送達日), status(pending/filed/missed), assigned_lawyer_id` | 時限主檔，**internal_date = statutory_date − buffer** |
| `hearings`（庭期） | `id, matter_id, court, hearing_at, hearing_type(言詞辯論/準備程序), assigned_lawyer_id` | 開庭 |
| `conflict_index`（利衝索引） | `matter_id, party_name, party_role(原告/被告/關係人), party_id_norm` | 利益衝突比對來源 |
| `billing`（計費/收款） | `id, matter_id, amount, due_date, status(unpaid/paid/overdue)` | 帳單到期 |

`deadlines` 的 `internal_date` 就是老闆說的「19 天」——詳見 §2.6 法律時限計算。

### 2.3 要新增哪些 cron job（頻率 + 資料流）

所有 cron job 鏡像 `flush_escalations.py` 的薄殼模式（獨立進程、`SME_FLOOR` 移除、讀 DB + token），各自只負責「掃 DB → enqueue row」，投遞交給既有三層。

| Cron job | 頻率 | 資料流 |
|---|---|---|
| **A. 時限掃描器** `scan_deadlines.py` | 每日一次（建議 `0 7 * * *`，早上 7:00） | SELECT `deadlines WHERE status='pending' AND internal_date <= today + N`（N=提醒前置天數，如 7）→ 對每筆 `enqueue_notification(kind='deadline_reminder', target=assigned_lawyer, summary='【案號X】上訴狀 內部期限 6/12（法定 6/13）剩 3 天')` → commit → 觸發三層投遞 |
| **B. 庭期掃描器** `scan_hearings.py` | 每日一次（`0 18 * * *`，前一日晚間） | SELECT `hearings WHERE hearing_at BETWEEN tomorrow_00 AND tomorrow_23` → `enqueue_notification(kind='hearing_reminder', summary='明日 6/12 09:30 台北地院 言詞辯論 案號X')` |
| **C. 收款掃描器** `scan_billing.py` | 每日 / 每週（`0 9 * * 1` 週一） | SELECT `billing WHERE status IN('unpaid') AND due_date <= today + N` 與 `status='overdue'` → `enqueue_notification(kind='billing_reminder')` |
| **D. 既有 flusher** `flush_escalations.py` | 每 2 分（既有） | 泛化為掃所有 `kind` 的 pending（保證層投遞主路，零改動只擴 SELECT 不限 `kind`） |

cron 安裝沿用 `install.sh:188-205` 的 crontab 寫法（marker 防重複 + WSL `cron daemon` 偵測）。

### 2.4 claude -p 在哪些點被 single-shot 呼叫（品質層）

`claude -p` 不負責「保證送達」（那是 cron + flusher 的事），它負責**把確定性的 row 寫成律師看得懂的人話 / 做需要判斷的工作**。三個 single-shot 呼叫點：

**(1) 每日時限摘要生成器**（時限掃描器 A 之後）
- 觸發：`scan_deadlines.py` enqueue 完一批 row 後，fire-and-forget 起 `claude -p`（鏡像 `spawn_notifier`：去 `ANTHROPIC_API_KEY` 走訂閱、去 `SME_FLOOR` 全權限、`SME_NOTIFIER=1` 等價防遞迴旗標、窄工具白名單 `list_pending_*` + `mcp__line__reply`）。
- 工作：把該律師當天所有到期 row 合併成**一則排序好的 LINE 摘要**（最急的在前、附法條依據與內部/法定兩個日期），而非每筆一則洗版。prompt 硬約束「收件人照 row、不同律師不合併」（沿用 `_NOTIFIER_PROMPT` 模式）。
- 兜底：保證層 cron 仍會逐筆推（claude -p 漏的/掛的）。

**(2) 接案利益衝突檢查器**（事件驅動，登錄新案時）
- 觸發：律師在互動 session 用 `create_matter` 登錄新案 → service in-tx 寫 `conflict_index` row + enqueue 一筆 `kind='conflict_check'` → commit 後起 `claude -p`。
- 工作：claude -p 讀新案所有當事人 → query 既有 `matters`/`conflict_index` 找對造重疊（同一人是 A 案的我方、B 案的對造＝利衝）、關係人交叉 → 有疑慮就 `reply` 通知負責合夥人 + 寫 `log_decision(confidential=True)` 留底。**這比純 SQL 強的地方**：人名變體（公司全名 vs 簡稱、自然人同名）、關係人間接關聯需要 LLM 判斷。
- 安全：仍走訂閱、窄工具、`conflict_index` 是機密軸（§機密軸：`store_fact(confidential=True)`）。

**(3) 收款催繳文案生成器**（收款掃描器 C 之後，選配 HITL）
- 觸發：`scan_billing.py` 標出逾期帳單後起 `claude -p` 草擬催款措辭。
- 工作：依逾期天數分級寫不同語氣（首催/二催/嚴正）的草稿。**對外催款屬行銷/對外訊息，須走 HITL `create_approval`**（不可 claude -p 直接發給客戶），草稿先推給負責律師核准。

### 2.5 對比與資料流總結圖

```
時間驅動（律所新增）              事件驅動（既有 escalation 複用）
─────────────────              ──────────────────────────
OS cron 每日掃 deadlines/         業務 service in-tx
hearings/billing                 enqueue_escalation
       │                                │
       ▼                                ▼
  enqueue 一批 pending row（含蓋章：收件人/來源/案號/法條）
       │
       ├──── 保證層：cron flush（每2分、純Python push、確定性必達）★律所命脈
       ├──── 品質層：claude -p single-shot（走訂閱、合併人話、利衝判斷）
       └──── 即時層：IPC 注入正在跑的律師 session（best-effort 即時自醒）
```

### 2.6 法律時限計算（已 WebSearch 查證、附法條）

**這是律所 vertical 的命脈，凡涉法定天數一律查證、不可憑印象。** `internal_date` 與 `statutory_date` 必須由確定性程式碼算（不可交給 LLM 算天數）。

| 程序 | 法定上訴期間 | 法條依據 |
|---|---|---|
| 民事二審上訴 | 判決送達後 **20 日**（不變期間） | 民事訴訟法第440條 |
| 刑事上訴 | 送達判決後 **20 日** | 刑事訴訟法第349條 |
| 行政訴訟上訴 | 高等行政法院判決送達後 **20 日** | 行政訴訟法第241條（向最高行政法院） |
| 訴願後提行政訴訟 | 訴願決定書送達次日起 **2 個月** | 行政訴訟法第106條 |

**計算規則（民法第119–124條，所有訴訟期間共用）**：
- **起算日不計、翌日起算**（民法第120條第2項）：判決送達日當天不算，從次日算第 1 日。
- **末日遇假日順延**（民法第122條）：末日為星期日/紀念日/休息日 → 順延至次一上班日。
- **在途期間**（民事訴訟法第162條）：當事人不在法院所在地、且無住法院所在地的訴訟代理人 → 加扣在途期間；**律師（訴訟代理人）住法院所在地則不加計**——律所多數情況律師在當地，故 system 預設不加在途，但 `deadlines` 應留 `in_transit_days` 欄供例外。

**老闆查到的「19 天」推斷已驗證為合理**：法定 20 日，內部期限 = 法定 − 安全緩衝。框架設計：`statutory_date` 由程式按民法第120/122條精算（含末日順延），`internal_date = statutory_date − buffer_days`（buffer 可設定，預設 1 天＝老闆的 19 天，可調更保守如 3 天）。提醒以 `internal_date` 為準推播，留出律師備狀緩衝。`scan_deadlines.py` 計算邏輯**必須有單元測試 + 對照司法院線上試算工具**，且天數錯誤＝執業過失，這段絕不可 LLM 即興算。

## 3. 為何律所更該走 system-layer（vs「互動 session 才會動」）

### 3.1 兩個模型的本質差異

| 面向 | 互動 session 模型 | system-layer 模型 |
|---|---|---|
| 觸發條件 | **人要打開 Claude 並問起** | **時間到自動發生**（cron） |
| 漏掉的後果 | 律師忙/休假沒開 → 時限靜默逼近 → 漏期＝過失 | 人沒開也在倒數、確定性推播 |
| 成本 | 每次互動都耗 session | cron 純 Python 零 LLM 成本；claude -p 走訂閱非 metered |
| 可靠性 | 取決於人的記性與習慣 | 保證層 cron + retry/backoff/failed 終態 |

### 3.2 為何律所特別契合

1. **時限是時間驅動、不是事件驅動**：法定 20 日從判決送達那刻就開始倒數，與「律師有沒有開 Claude」完全無關。把它綁在互動 session ＝把「絕不能漏」的事綁在「人可能不做」的前提上，方向性錯誤。
2. **漏一個＝執業過失**（不可逆、賠償+懲戒）：這正是既有 escalation「保證層 cron 確定性必達 + retry/failed 終態 + 開機 readout 露 failed/無收件人」存在的理由。律所時限是比超門檻記帳更該享受這套保證的場景。
3. **每天固定節律**（早上看今天到期、前一晚看明天庭期）天生是 cron 排程，不是按需查詢。
4. **既有基礎設施現成**：三層投遞、防雙送租約、蓋章不重算、走訂閱不 metered——律所框架几乎只需新增「cron 掃描 → enqueue」寫入端，投遞層原樣複用。

### 3.3 哪些事仍必須在互動 session 做

system-layer 負責「**提醒/監看/確定性投遞**」，**判斷/輸入/不可逆動作仍在互動 session**：

- **登錄案件與起算事件**：判決送達日、開庭通知日由律師/助理在 session 輸入（cron 才有資料可算 `statutory_date`）。送達日是人工事實認定，機器無法自動知道。
- **確認/標記時限已完成**：律師遞交書狀後在 session `mark_deadline_filed`（status `pending→filed`），否則 cron 會一直提醒。
- **利衝最終裁決**：claude -p 只**標記疑似**利衝；是否接案、是否設資訊隔離牆（Chinese Wall）由合夥人在 session 決定 + `log_decision` 留底。
- **對外催款 / 任何對客戶訊息**：須走 HITL `create_approval`（CLAUDE.md HITL 規則），claude -p 只草擬、不直接發。
- **時限例外計算**：在途期間加計、回復原狀聲請（民訴第164條：天災等不可歸責事由、原因消滅後10日內、逾1年不得聲請）等需人工判斷的，由律師在 session 處理。
- **規則/SOP 寫入**：律所內規（如「內部緩衝統一抓 3 天」）由老闆在 session 經知識庫寫入 flow 落 `store_fact`。
